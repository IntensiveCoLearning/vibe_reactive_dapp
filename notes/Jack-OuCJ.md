# **Reactive Network 学习笔记0310**

## **1\. Reactive 合约内部机制**

创建睿应式合约时需要指定：

-   监听的 **区块链**
    
-   监听的 **合约地址**
    
-   监听的 **事件（topic0）**
    

执行过程：

`监听目标事件 ↓ 检测到事件触发 ↓ 执行合约逻辑 ↓ 更新合约状态 ↓ 发起新的链上交易`

Reactive 合约是 **有状态的**，可以：

-   累积历史事件数据
    
-   根据历史 + 新事件组合触发条件
    

## **2\. 睿应层适合解决什么问题？**

**凡是需要"链上状态变化 → 自动触发操作"的场景，都适合用睿应层。**

| 场景 | 传统做法 | 睿应做法 |
| --- | --- | --- |
| DEX 止损单 | 机器人监控价格，持私钥执行 | 监听 Uniswap swap 事件，价格触达自动执行 |
| DEX 套利 | Bot 轮询多池价差 | 监听多池价格差异，自动套利（支持跨链） |
| 多预言机聚合 | 链下服务定时拉数据 | 监听多个预言机更新事件，链上聚合计算 |
| 流动性再平衡 | 人工或中心化服务监控 | 监听多链流动性状况，条件满足自动再平衡 |

# Reactive Network 学习笔记0309

## 一、睿应层（Reactive Network）vs 传统 EVM：根本区别

### 传统智能合约

-   **被动执行**：只有被人（EOA）调用才会运行
    
-   **无自主感知**：合约本身感知不到链上发生了什么
    
-   **自动化依赖外部**：需要链下 Bot/Keeper 不断轮询，存在中心化风险
    

```
传统模式：
用户/机器人 → 发起交易 → 合约执行
          ↑
     需要私钥 + 链下服务维持
```

### 睿应式合约（Reactive Contract）

-   **主动感知**：持续监听指定链上的事件（Event）
    
-   **自动触发**：检测到预设事件 → 自动执行逻辑，无需任何人工干预
    
-   **完全链上**：输入是链上事件，输出是链上操作，全程去中心化
    

```
睿应模式：
链上事件发生 → 睿应层捕获 → 合约自动执行 → 链上回调
（无需机器人，无私钥托管，无中心化依赖）
```

* * *

## 二、核心架构：ReactVM + 睿应层

### 架构分层

```
┌──────────────────────────────────┐
│         目标链（ETH/其他EVM）      │  ← 事件来源 / 回调目标
├──────────────────────────────────┤
│         睿应层（Reactive Network）│  ← 监听、路由、触发执行
│  ┌────────────────────────────┐  │
│  │      ReactVM（执行环境）    │  │  ← 运行睿应式合约逻辑
│  └────────────────────────────┘  │
└──────────────────────────────────┘
```

### 关键概念

| 概念 | 说明 |
| --- | --- |
| ReactVM | 专门运行睿应式合约的 EVM 沙箱环境 |
| 睿应层 | 监听多链事件、路由触发、协调跨链执行的基础设施 |
| 控制反转 | 合约不再等待被调用，而是主动响应事件 |
| 有状态执行 | 合约可积累历史事件数据，满足条件后再触发操作 |

* * *

## 三、控制反转（IoC）——核心思想

> 传统：**"你来叫我，我才动"**  
> 睿应：**"我自己盯着，条件到了我自动动"**

**解决的本质问题**：

-   不再需要中心化机器人持有私钥
    
-   不再依赖链下服务 7×24 轮询
    
-   逻辑完全在链上，可审计、不可篡改
