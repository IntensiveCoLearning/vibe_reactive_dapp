关于 Reactive Contract 组成结构的学习笔记

最近在学习 Reactive Contract（反应式合约）的时候，一开始其实是有点懵的。因为它和传统的智能合约思路不太一样。传统智能合约大多数时候是“被调用才执行”，而 Reactive Contract 更像是一种“监听—触发—执行”的机制。慢慢理解之后，会发现它的结构其实挺有逻辑的，大致可以拆成几个核心部分。

## 一、Reactive Contract 的整体思路

先从整体理解。Reactive Contract 本质上就是一种能够 **根据外部事件自动做出反应的智能合约**。  
换句话说，它不是一直等别人调用，而是会主动“关注”某些事情，一旦这些事情发生，就触发合约逻辑。

所以从结构上看，它一定需要解决三个问题：

1.  **监听什么事情（Event / Trigger）**
    
2.  **什么时候触发（Condition）**
    
3.  **触发之后做什么（Action / Execution）**
    

这三点基本就构成了 Reactive Contract 的核心逻辑框架。

## 二、事件来源（Event Source）

Reactive Contract 的第一部分，就是 **事件来源**。

因为合约要“反应”，就必须先知道 **它在监听什么**。这些事件一般来自几个地方：

-   **区块链上的交易或事件**
    
-   **其他智能合约的状态变化**
    
-   **跨链消息**
    
-   **外部数据源（比如 Oracle）**
    

举个简单的例子：

比如有一个合约在监听：

-   某个地址收到资金
    
-   某个 NFT 被交易
    
-   某条跨链消息到达
    

当这些事件发生的时候，Reactive Contract 就会收到信号，然后判断是否要执行下一步逻辑。

所以从结构上看，Reactive Contract 需要定义 **监听对象和监听规则**。

## 三、触发条件（Trigger Condition）

有了事件来源，还不够，因为不是所有事件都会触发执行。

所以 Reactive Contract 还有一个非常关键的部分，就是 **触发条件**。

也就是说：

> 当监听到事件之后，还要再判断一次条件是否成立。

比如：

-   金额是否超过某个值
    
-   交易是否来自特定地址
    
-   时间是否达到某个区块高度
    
-   某个状态变量是否改变
    

只有当这些条件满足时，合约才会进入执行阶段。

这一层其实很像现实生活中的 **“如果…那么…”逻辑**。

例如：

如果  
某地址转入 ≥ 10 ETH  
那么  
执行自动分配资金。

这种条件判断是 Reactive Contract 自动化的重要基础。

## 四、执行逻辑（Action / Execution）

当事件发生，并且条件满足之后，Reactive Contract 就会进入 **执行阶段**。

执行阶段其实就是普通智能合约的逻辑部分，比如：

-   调用其他合约
    
-   发送资产
    
-   更新状态变量
    
-   触发新的事件
    
-   发起跨链操作
    

例如：

当监听到某 NFT 成交时：

-   自动分配版税
    
-   自动触发二级市场交易
    
-   自动进行跨链同步
    

这一部分本质上就是 **合约的业务逻辑**，只是触发方式变成了自动化。

## 五、状态管理（State）

除了监听和执行之外，Reactive Contract 还需要管理自己的 **状态（State）**。

因为很多反应逻辑都依赖于历史信息，比如：

-   是否已经执行过某个操作
    
-   当前任务是否完成
    
-   某个条件是否已经被触发
    

如果没有状态管理，合约就可能：

-   重复执行
    
-   出现逻辑冲突
    
-   无法记录执行过程
    

所以 Reactive Contract 也会像普通智能合约一样维护一些状态变量，用来保证逻辑正确。

## 六、总结：Reactive Contract 的结构

从整体上看，Reactive Contract 的组成结构可以理解为一个简单的逻辑链：

**事件监听 → 条件判断 → 自动执行 → 状态更新**

换一种更直白的理解方式：

Reactive Contract 就像一个 **自动值班的程序**：

-   一直在观察某些事情
    
-   一旦发生特定事件
    
-   判断条件是否满足
    
-   然后自动执行设定好的操作
    

相比传统智能合约，这种模式更接近 **自动化系统**，也更适合做跨链交互、自动清算、自动交易等场景。

## 七、个人理解

我自己感觉 Reactive Contract 最大的意义，其实在于把区块链应用从 **“被动执行”变成“主动响应”**。

传统智能合约像一个工具：

有人调用 → 它才工作。

而 Reactive Contract 更像一个机器人：

它会自己盯着事情，一旦发生就立刻行动。

如果未来跨链和自动化应用越来越多，这种模式应该会越来越重要。

所以理解它的组成结构，本质上就是理解这四件事：

**监听什么 → 什么时候触发 → 触发后做什么 → 如何记录状态。**

把这四点搞清楚，Reactive Contract 的整体逻辑基本也就明白了。

Reactive Network 跨链事件回调 — 完整实战总结

一、任务背景

构建一个跨链事件与回调合约 — 使用 Reactive Network 的 Basic Demo，在 Sepolia 上部署 Origin 和 Callback 合约，在 Reactive Lasna 测试网上部署 Reactive 合约，跑通整条事件监听→回调链路。

二、核心原理

2.1 Reactive Network 是什么

Reactive Network 是一个 EVM 自动化层。它的核心能力是：让智能合约可以「监听」任意 EVM 链上的事件，用 Solidity 逻辑做判断，然后自动触发跨链交易。不需要 bot，不需要链下基础设施，全链上的 IFTTT。

2.2 三合约架构

整个 Basic Demo 由三个合约构成，各自部署在不同网络：

Origin 合约 (BasicDemoL1Contract.sol) → 部署在 Sepolia

• 功能极其简单：接收 ETH，退还给发送者，同时发出一个 Received 事件

• 事件签名（topic\_0）：0x8cabf31d2b1b11ba52dbb302817a3c9c83e4b2a5194d35121ab1354d69f6a4cb

Reactive 合约 (BasicDemoReactiveContract.sol) → 部署在 Reactive Lasna

• 在构造函数中调用系统合约的 subscribe() 方法，订阅 Origin 合约在 Sepolia 上的 Received 事件

• 当监听到事件后，检查 topic\_3（金额）是否 ≥ 0.01 ETH

• 条件满足时，发出一个 Callback 事件，携带 payload 去调用 Destination 合约的 callback() 函数

Destination/Callback 合约 (BasicDemoL1Callback.sol) → 部署在 Sepolia

• 接收来自 Reactive Network 的回调

• 通过 authorizedSenderOnly 修饰器确保只有授权的回调代理地址可以调用

• 执行回调后发出 CallbackReceived 事件

2.3 数据流

你发 0.01 ETH → Origin合约(Sepolia)

↓ 发出 Received 事件

Reactive合约(Lasna) 监听到

↓ 检查金额 ≥ 0.01 ETH ✓

↓ 发出 Callback 事件

Callback Proxy(Sepolia) 转发

↓

Destination合约(Sepolia) 收到回调

↓ 发出 CallbackReceived 事件 ✅

2.4 关键概念

• subscribe() — Reactive 合约在构造时调用系统合约（0x...fffFfF）的 subscribe 方法，注册要监听的 chainId + 合约地址 + event topic

• topic\_0 — 事件的函数签名哈希，用于精确匹配要监听的事件类型

• Callback Proxy — Reactive Network 在每条支持的链上部署的代理合约，负责将 Reactive 的回调指令转为链上交易

• authorizedSenderOnly — Callback 合约的权限校验，确保只接受来自 Callback Proxy 的调用

五、遇到的问题与解决

问题 1：cat > 写文件报错 "No such file or directory"

原因：目标目录 script/ 不存在

解决：mkdir -p script 先建目录再写文件。cat > 只能写文件，不会自动创建目录。

问题 2：环境变量未设置

原因：SEPOLIA\_RPC、ORIGIN\_ADDR、CALLBACK\_ADDR 等变量没有 export

解决：部署前必须逐一 export 所有需要的变量。用 echo $变量名 验证。

问题 3：公共 RPC 522 超时

原因：[rpc.sepolia.org](http://rpc.sepolia.org) 不稳定，经常超时

解决：换成 [https://ethereum-sepolia-rpc.publicnode.com。长期建议用](https://ethereum-sepolia-rpc.publicnode.com。长期建议用) Alchemy/Infura 的免费 API。

问题 4：forge create 只模拟不上链

原因：缺少 --broadcast 参数

解决：forge create 默认是 dry run（模拟），必须加 --broadcast 才会真正发交易上链。这是一个安全设计。

问题 5：Callback 合约部署缺少构造参数

报错：no arguments provided for contract constructor

原因：BasicDemoL1Callback 的构造函数需要一个 _callback_sender 地址参数

解决：加 --constructor-args $DESTINATION\_CALLBACK\_PROXY\_ADDR。这个地址是 Reactive Network 在 Sepolia 上的回调代理地址。

问题 6：--constructor-args 参数位置错误

报错：the following required arguments were not provided: <CONTRACT>

原因：--constructor-args 后面的内容会被当成参数的一部分，导致合约路径被吞掉

解决：合约路径放在 --constructor-args 之前，或者把 --constructor-args 及其值放在命令最后。

问题 7：Reactive 合约 subscribe 调用 revert

报错：subscribe → Revert → Failure

原因：最初用自写的 DeployReactive.s.sol 脚本部署，参数和官方要求不完全匹配

解决：改用官方文档的 forge create 命令直接部署，传入正确的 6 个构造参数，部署成功。

问题 8：最初 Callback 合约参数传错

原因：第一次部署 Callback 时用了 0x...FFFFFF（系统合约地址），但正确的应该是 DESTINATION\_CALLBACK\_PROXY\_ADDR（0xc9f36411...）

解决：重新部署 Callback 合约，传入正确的 Callback Proxy 地址。旧的合约在链上仍然存在但不会被使用。

六、关键要点总结

1\. 严格按官方 README 走 — 项目自带详细的部署命令，不需要手写部署脚本

2\. 环境变量是基础 — 部署前 export 全部变量，用 echo 验证

3\. RPC 不稳定要有备选 — 公共 RPC 经常挂，准备 2-3 个备选或用 Alchemy

4\. --broadcast 不能忘 — forge 默认模拟，不加就白跑

5\. 构造参数要精确 — Callback 合约需要 Proxy 地址，Reactive 合约需要 6 个参数

6\. 参数顺序要注意 — --constructor-args 放在合约路径后面最安全

7\. 验证用区块浏览器 — Sepolia Etherscan 查 Events 是最直观的验证方式

Reactive Network（睿应层）vs 传统 EVM：根本区别

一、核心范式差异：控制反转（Inversion of Control）

这是理解 Reactive Network 最关键的一个概念。

传统 EVM 智能合约是被动的——它们只有在外部账户（EOA）发起交易时才会执行。没人调用，合约就是一段沉默的代码。要实现"自动化"，你需要在链下跑 bot、keeper、cron job 来轮询链上状态然后触发交易。

Reactive Contract（RC）是主动的——它们订阅链上事件，当事件发生时自动执行预定义的 Solidity 逻辑。不需要任何人签交易，不需要链下 bot。合约自己决定什么时候执行。

用一句话概括：传统合约是"被叫才动"，RC 是"看到就动"。

二、架构差异

传统 EVM：

• 单链部署，单一状态

• 用户/bot 发 tx → 合约执行 → 改变状态

• 要跨链就得靠桥、Relayer、外部基础设施

Reactive Network 的三层架构：

1\. Origin Chain（来源链）：任何 EVM 链，RC 从这里监听事件日志

2\. ReactVM（执行环境）：私有的并行化 EVM 实例，RC 在这里处理事件、运行逻辑

3\. Destination Chain（目标链）：RC 通过 Callback 向这里发送交易

一个 RC 部署后实际存在于两个环境中（相同字节码，独立状态）：

• Reactive Network 上的实例：管理订阅、接收 EOA 调用（比如管理员操作）

• ReactVM 中的实例：处理事件、执行自动化逻辑

三、执行模型对比

| 维度 | 传统 EVM | Reactive Network |

| ----- | ------------- | ------------------ |

| 触发方式 | EOA 发交易 | 链上事件自动触发 |

| 跨链能力 | 需外部桥/Relayer | 原生跨链事件监听+回调 |

| 自动化依赖 | 链下 bot/keeper | 完全链上，去中心化 |

| 执行环境 | 共享全局 EVM | 隔离的 ReactVM（可并行） |

| 状态模型 | 单一状态 | 双状态（RNK + ReactVM） |

四、核心机制详解

1\. 订阅（Subscription）

RC 声明要监听哪条链、哪个合约、哪个事件（topic\_0）。Reactive Network 持续扫描这些链的事件日志。

2\. react() 方法

当订阅的事件发生时，网络自动调用 RC 的 react(LogRecord) 方法，传入完整的事件信息（chain\_id、合约地址、topics、data、区块号等）。

3\. Callback 事件

RC 处理完逻辑后，如果需要在目标链上执行操作，emit 一个 Callback 事件：

event Callback(

uint256 indexed chain\_id, // 目标链

address indexed \_contract, // 目标合约

uint64 indexed gas\_limit, // gas 限制

bytes payload // 编码的函数调用

);

Reactive Network 捕获这个 Callback，在目标链上提交对应交易。

五、典型应用场景

• DEX 止损/止盈单：监听 Uniswap 池的 Swap 事件，价格到阈值自动执行

• 清算保护：监听借贷协议健康因子，自动补充抵押或还款

• 跨链套利：同时监听多链 DEX 价格差，自动套利

• 流动性再平衡：多链流动性池自动调配资金

• 跨链 NFT/奖励分发：A 链上的游戏行为自动触发 B 链上的奖励

• 自动化费用收集：协议收入自动汇总、转换、分发

## 一、项目介绍

### 1.1 什么是 Reactive Network？

Reactive Network（睿应层）是 Web3 首个去中心化执行层，让部署在同链或跨链的互不相关合约实现响应式交互。

简单来说：它是一个完全链上、事件驱动的「if-this-then-that（条件触发）」网络，用于去中心化自动化，并简化工作流。

### 1.2 核心特点

-   **事件驱动**：合约可以监听其他链上的事件并自动执行
    
-   **跨链自动化**：实现 Origin → Reactive → Destination 的自动化流程
    
-   **条件触发**：支持设置条件判断，满足条件才执行回调
    
-   **无需中心化服务**：完全去中心化，不依赖中心化 relayer

## 三、核心概念

### 3.1 Subscribe-Trigger-Callback 模型

```

┌─────────────┐     事件      ┌─────────────┐    条件判断    ┌─────────────┐

│    Origin   │ ──────────▶  │   Reactive  │ ──────────▶  │ Destination │

│   (源链)    │              │   (中间层)  │              │   (目标链)  │

└─────────────┘              └─────────────┘              └─────────────┘

  触发事件                       监听事件                     执行回调

                                判断条件

```

### 3.2 三段链路详解

| 环节 | 角色 | 说明 |

|------|------|------|

| Origin | 源合约 | 部署在源链上，当发生特定行为时 emit 事件 |

| Reactive | 中间层合约 | 部署在 Reactive Network 上，订阅 Origin 的事件，检测条件是否满足 |

| Destination | 目标合约 | 部署在目标链上，接收 Reactive 的回调，执行最终业务逻辑 |

### 3.3 适用场景

-   自动化止盈止损订单
    
-   清算保护
    
-   自动化投资组合再平衡
    
-   收益优化
    
-   跨链工作流
    
-   链上提醒通知
    
-   NFT 地板价监控

### 4.2 OriginContract.sol（源合约）

```solidity

// SPDX-License-Identifier: MIT

pragma solidity ^0.8.19;

  

contract OriginContract {

    event TriggerEvent(

        address indexed user,

        uint256 amount,

        string message

    );

  

    mapping(address => uint256) public lastAmount;

  

    function triggerAction(uint256 amount, string memory message) external {

        require(amount > 0, "Amount must be greater than 0");

        lastAmount[msg.sender] = amount;

        emit TriggerEvent(msg.sender, amount, message);

    }

}

```

### 4.3 ReactiveContract.sol（Reactive 中间层）

```solidity

// SPDX-License-Identifier: MIT

pragma solidity ^0.8.19;

  

import {Reactive} from "reactive/abstract/Reactive.sol";

import {IReactive} from "reactive/interfaces/IReactive.sol";

  

contract ReactiveContract is Reactive {

    event ReceivedEvent(

        uint256 srcChainId,

        address srcContract,

        address user,

        uint256 amount

    );

  

    uint256 public destinationChainId;

    address public destinationContract;

    uint256 public minAmount;

  

    constructor(

        uint256 _destinationChainId,

        address _destinationContract,

        uint256 _minAmount

    ) {

        destinationChainId = _destinationChainId;

        destinationContract = _destinationContract;

        minAmount = _minAmount;

    }

  

    function setMinAmount(uint256 _minAmount) external onlyOwner {

        minAmount = _minAmount;

    }

  

    function subscribe(

        uint256 chainId,

        address originContract,

        bytes32 eventSignature,

        uint256 interval

    ) external onlyOwner {

        react.subscribe(chainId, originContract, eventSignature, interval);

    }

  

    function react(

        uint256 srcChainId,

        address srcContract,

        bytes calldata data

    ) external override onlyReactor {

        (address user, uint256 amount, string memory message) = abi.decode(

            data,

            (address, uint256, string)

        );

  

        emit ReceivedEvent(srcChainId, srcContract, user, amount);

  

        if (amount >= minAmount) {

            react.callback(

                destinationChainId,

                destinationContract,

                abi.encode(user, amount, message)

            );

        }

    }

}

```

### 4.4 CallbackContract.sol（目标回调合约）

```solidity

// SPDX-License-Identifier: MIT

pragma solidity ^0.8.19;

  

import {IReactiveCallback} from "reactive/interfaces/IReactiveCallback.sol";

  

contract CallbackContract is IReactiveCallback {

    event CallbackReceived(

        address user,

        uint256 amount,

        string message,

        uint256 timestamp

    );

  

    mapping(address => uint256[]) public userAmounts;

    address[] public users;

  

    function receiveCrossChainCallback(

        uint256 srcChainId,

        address srcContract,

        bytes calldata data

    ) external override {

        (address user, uint256 amount, string memory message) = abi.decode(

            data,

            (address, uint256, string)

        );

  

        userAmounts[user].push(amount);

        if (userAmounts[user].length == 1) {

            users.push(user);

        }

  

        emit CallbackReceived(user, amount, message, block.timestamp);

    }

  

    function getUserHistory(address user) external view returns (uint256[] memory) {

        return userAmounts[user];

    }

}

```

### 4.5 部署脚本 Deploy.s.sol

```solidity

// SPDX-License-Identifier: MIT

pragma solidity ^0.8.19;

  

import "forge-std/Script.sol";

import {OriginContract} from "../src/OriginContract.sol";

import {ReactiveContract} from "../src/ReactiveContract.sol";

import {CallbackContract} from "../src/CallbackContract.sol";

  

contract DeployScript is Script {

    function run() external {

        uint256 deployerPrivateKey = vm.envUint("REACTIVE_LASNA_PRIVATE_KEY");

        vm.startBroadcast(deployerPrivateKey);

  

        // Chain IDs (Testnet)

        // Ethereum Sepolia: 11155111

        // Base Sepolia: 84532

        // Reactive Lasna: 5318007

  

        // 1. 部署 CallbackContract 到目标链（Base Sepolia）

        CallbackContract callback = new CallbackContract();

        console.log("CallbackContract deployed at:", address(callback));

  

        // 2. 部署 ReactiveContract 到 Reactive Lasna

        ReactiveContract reactive = new ReactiveContract(

            84532,                      // destinationChainId (Base Sepolia)

            address(callback),          // destinationContract

            100                         // minAmount

        );

        console.log("ReactiveContract deployed at:", address(reactive));

  

        // 3. 部署 OriginContract 到源链（Ethereum Sepolia）

        OriginContract origin = new OriginContract();

        console.log("OriginContract deployed at:", address(origin));

  

        // 4. 订阅事件

        reactive.subscribe(

            11155111,                  // sourceChainId (Ethereum Sepolia)

            address(origin),            // originContract

            bytes32(

                keccak256("TriggerEvent(address,uint256,string)")

            ),

            0                           // interval (0 = trigger on every event)

        );

        console.log("Subscribed to TriggerEvent");

  

        vm.stopBroadcast();

    }

}

```

## 十二、附录：常见问题

### Q1: 什么是 Reactive Contract？

A: Reactive Contract（响应式智能合约）是部署在 Reactive Network 上的智能合约，它可以监听其他链上的事件，并根据预设条件自动触发跨链回调。

### Q2: Reactive 和传统方案有什么区别？

A: 传统方案需要中心化的 bot 或服务轮询链上数据，而 Reactive 是完全去中心化的事件驱动架构，合约自动监听和执行。

### Q3: 哪些链可以作为 Origin/Destination？

A: 详见官方文档 Origins & Destinations 页面。主网支持 Ethereum、Arbitrum、Avalanche、Base、BSC 等；测试网支持 Ethereum Sepolia、Base Sepolia 等。

### Q4: 如何调试 Reactive 合约？

A: 使用 Reactscan ([https://reactscan.net](https://reactscan.net)) 查看链上交易和事件日志。
