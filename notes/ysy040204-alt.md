# Reactive Network 经济模型笔记

Reactive Network 的经济模型本质 = Gas 驱动 + 预付费 + 自动结算 + 跨链回调计费系统

它解决的是：

**谁为“自动执行 + 跨链回调”买单？**

# 费用体系（最核心）

## 1️⃣ Reactive Transaction Fee（反应交易费用）

公式：

fee=BaseFee⋅GasUsedfee = BaseFee \\cdot GasUsedfee=BaseFee⋅GasUsed

### 解释：

**BaseFee**：区块基础 gas 价格（类似 EIP-1559）

**GasUsed**：实际执行消耗 gas

本质和以太坊一样：**用多少算多少 × 当前网络价格**

## 关键特点

费用 **不是提前固定**

是 **执行后按真实 gas 结算**

计费发生在：

区块 n 执行

区块 n+1 结算

类似：**“先跑程序 → 再扣钱”**

# 支付机制

Reactive Network 采用：

**预充值 + 债务模型（Debt Model）**

## 两种充值方式

### 方式一：直接转账

```
cast send CONTRACT_ADDR --value 0.1ether
```

然后必须：

```
coverDebt()
```

手动清债

* * *

### 方式二：系统合约充值（推荐）

```
depositTo(address)
```

特点：

自动结算 debt

不需要手动调用 coverDebt

实战中几乎都用这个

核心逻辑总结：

| 步骤 | 行为 |
| --- | --- |
| 1 | 给合约充值 |
| 2 | 执行 reactive 逻辑 |
| 3 | 系统记录费用（Debt） |
| 4 | 自动/手动结算 |

## Debt（债务机制）

Reactive Network 有一个非常重要的概念：**Contract Debt（合约欠费）**

## 如果不付钱会怎样？

直接后果：被 blocklist（黑名单）不再执行回调 合约变 inactive

本质理解Reactive 是：

**“先执行 → 记账 → 再扣钱” 的信用系统**

# Callback 经济模型（跨链重点🔥）

跨链执行 = 核心价值

## Callback 特点

也是 **预付费机制**

和 reactive transaction 一样

不付钱 → 不执行

# Subscriptions 的作用

**Subscription（订阅）是 Reactive Network 的核心机制之一。**

它的作用是：

> 让 Reactive Contract 自动监听某些区块链事件，当事件发生时自动执行逻辑。

Reactive Contract 不需要用户触发交易，而是 **自动响应链上事件**。

例如：

| 事件 | 自动执行 |
| --- | --- |
| Uniswap 价格变化 | 自动止损 |
| 借贷清算 | 自动补仓 |
| token transfer | 自动处理逻辑 |

这种模式类似：

```
If Event Happens
      ↓
Reactive Contract Executes Logic
```

也可以理解为：

**Smart Contract 的 IFTTT（If This Then That）自动化系统**

* * *

# Reactive Contract 如何订阅事件

Reactive Contract 通过调用 **Reactive Network 的 System Contract** 来订阅事件。

使用函数：

```
subscribe()
```

通常在 **constructor()** 中设置订阅。

* * *

## 示例代码

```
service.subscribe(
    originChainId,
    _contract,
    _topic_0,
    REACTIVE_IGNORE,
    REACTIVE_IGNORE,
    REACTIVE_IGNORE
);
```

意思是：

订阅某条链某个合约的某个事件。

* * *

# subscribe() 函数参数解析

函数定义：

```
function subscribe(
    uint256 chain_id,
    address _contract,
    uint256 topic_0,
    uint256 topic_1,
    uint256 topic_2,
    uint256 topic_3
)
```

参数解释：

| 参数 | 含义 |
| --- | --- |
| chain_id | 事件来源链 |
| _contract | 触发事件的合约 |
| topic_0 | event signature |
| topic_1 | indexed event 参数 |
| topic_2 | indexed event 参数 |
| topic_3 | indexed event 参数 |

Reactive Network 可以根据这些字段过滤事件。

* * *

# 事件过滤机制（Filtering）

Reactive 支持 **精确事件过滤**。

可以按以下条件订阅：

chain id  
contract address  
event topic0  
event topic1  
event topic2  
event topic3

例如：

监听

```
Uniswap Pair → Sync Event
```

过滤条件：

```
chainId = Ethereum
contract = Uniswap Pair
topic0 = Sync
```

* * *

# REACTIVE\_IGNORE（通配符）

如果某些字段不需要过滤，可以使用：

```
REACTIVE_IGNORE
```

示例：

```
subscribe(
chainId,
contract,
topic0,
REACTIVE_IGNORE,
REACTIVE_IGNORE,
REACTIVE_IGNORE
)
```

意思：

只监听 topic0。

* * *

# 静态订阅 vs 动态订阅

Reactive 支持两种订阅方式。

* * *

# 静态订阅（Static Subscription）

在 **constructor()** 中设置。

示例：

```
constructor() {
   subscribe(...)
}
```

特点：

部署时就确定

最常见

* * *

# 动态订阅（Dynamic Subscription）

Reactive Contract 运行过程中可以：

```
subscribe()
unsubscribe()
```

动态添加或删除订阅。

通常通过：

```
Callback
```

机制实现。

* * *

# unsubscribe()

取消订阅函数：

```
unsubscribe()
```

结构和 subscribe 一样。

```
function unsubscribe(
    chain_id,
    contract,
    topic0,
    topic1,
    topic2,
    topic3
)
```

用途：

停止监听某些事件

减少 gas 消耗

* * *

# react() 事件处理流程

当 Reactive Network 发现符合条件的事件时：

Reactive Contract 的

```
react()
```

函数会被触发。

示例：

```
function react(LogRecord calldata log) external vmOnly
```

事件数据包括：

```
chain_id
contract
topic0
topic1
topic2
topic3
data
```

react() 可以：

更新状态

计算逻辑

触发 callback transaction

* * *

# Callback（跨链执行）

Reactive Contract 可以发起 callback：

```
emit Callback(
    chain_id,
    contract,
    gas_limit,
    payload
);
```

Reactive Network 会：

监听 Callback event  
在 destination chain 发交易

实现：

```
Cross-chain automation
```

例如：

```
Uniswap price < threshold
→ send transaction
→ sell token
```

# Reactive Network + Uniswap V2 + 预言机 + 止损单案例

* * *

> **让合约不再被动等人调用，而是主动监听链上事件，并在条件满足时自动执行链上操作。**

传统智能合约的问题是：

-   只能被 EOA 或其他合约调用
    
-   自己不能主动“感知外界”
    
-   不能定时执行
    
-   不能直接访问链下世界
    

Reactive Network 的意义就是：

-   监听链上事件
    
-   在 ReactVM 中自动执行逻辑
    
-   条件满足后发起回调交易
    
-   从而实现自动化、跨链、事件驱动的工作流
    

* * *

# 先理解传统智能合约为什么不够

## 1\. 传统合约是“被动”的

普通智能合约不能自己运行。  
它必须等：

-   用户钱包发交易
    
-   或别的合约在交易里调用它
    

所以即使你写了一个函数：

```
function getLatestPrice() public view returns (int)
```

它也不会自己每秒跑一次。

* * *

## 2\. 这会导致很多自动化场景很难做

例如：

-   价格跌破阈值自动止损
    
-   抵押率不够自动清算
    
-   监控别的链上事件自动跟单
    
-   某个事件发生后自动触发跨链结算
    

传统做法只能靠链下脚本：

-   Python 机器人
    
-   keeper 服务
    
-   cron job
    
-   后端轮询
    

问题：

-   要一直在线
    
-   有延迟
    
-   有单点故障
    
-   不够去中心化
    
-   安全边界差
    

* * *

# 预言机解决什么问题

## 1\. 区块链本身看不到链下世界

智能合约运行在一个封闭系统里，它不能直接访问：

-   网站 API
    
-   数据库
    
-   直播平台
    
-   天气
    
-   比赛结果
    
-   现实价格
    

所以需要预言机把链下数据送上链。

* * *

## 2\. 预言机不是数据源，而是数据搬运和验证层

预言机的职责：

-   从外部抓数据
    
-   聚合多个来源
    
-   验证真实性
    
-   签名或共识后提交到链上
    

比如 Chainlink、Band Protocol。

* * *

## 3\. 传统预言机的局限

即使预言机把数据上链了，合约还是不会自己动。  
因为：

-   合约不会主动轮询价格
    
-   状态变化仍然需要一笔交易触发
    

所以只靠预言机还不够。

* * *

# Reactive Network 的核心意义

Reactive Network 把“预言机上链的数据”变成“自动触发的动作”。

完整思路：

1.  链下世界变化
    
2.  预言机把结果上链
    
3.  链上发出事件
    
4.  Reactive Contract 监听到事件
    
5.  在 ReactVM 中执行逻辑
    
6.  触发回调交易
    
7.  在目标链执行动作
    

所以可以记成：

> **链下数据 → 预言机上链 → 事件 → Reactive 合约 react() → Callback → 自动执行**

* * *

# Reactive Network 的两个运行环境

这是今天最重要的概念之一。

一个响应式合约虽然只有一份代码，但实际上会在 **两个环境** 里各运行一份实例。

* * *

## 1\. Reactive Network（RN）

可以理解成：

-   前台
    
-   大厅
    
-   主网络环境
    

职责：

-   管理订阅
    
-   接收用户显式交易
    
-   与系统合约交互
    
-   转发事件到 ReactVM
    
-   接收 ReactVM 发出的 Callback 请求
    

* * *

## 2\. ReactVM

可以理解成：

-   实验室
    
-   引擎
    
-   隔离沙盒
    

职责：

-   接收来自 RN 转发的事件
    
-   运行 `react()`
    
-   执行业务判断
    
-   发出 `Callback` 请求
    

特点：

-   外部用户不能直接调用
    
-   只能由系统驱动
    
-   同部署者地址隔离
    
-   更适合高并发逻辑处理
    

* * *

## 3\. 一定要记住：同一份代码，两个实例，状态独立

也就是说：

-   RN 里有一套状态
    
-   ReactVM 里也有一套状态
    

变量名一样，不代表状态共享。

比如：

```
bool triggered;
```

在 RN 和 VM 中概念上是两份不同的值。

* * *

# 如何判断当前在 RN 还是 ReactVM

Reactive Network 用一个特殊系统地址来帮助判断环境：

```
0x0000000000000000000000000000000000fffFfF
```

判断逻辑：

-   在 RN 中，这个地址有代码
    
-   在 ReactVM 中，这个地址没有代码
    

所以合约用 `extcodesize` 检测：

-   有代码 → RN
    
-   没代码 → ReactVM
    

最终得到一个布尔值：

```
vm
```

含义：

-   `vm == false` → 当前在 RN
    
-   `vm == true` → 当前在 ReactVM
    

* * *

# 两个重要修饰符

因为不是所有函数都该在两个环境里执行，所以要加门禁。

* * *

## 1\. `rnOnly`

```
modifier rnOnly() {
    require(!vm, "Reactive Network only");
    _;
}
```

作用：

-   只允许在 RN 执行
    
-   常用于管理函数、订阅函数
    

比如：

-   `pause()`
    
-   `resume()`
    
-   `subscribe()`
    
-   `unsubscribe()`
    

* * *

## 2\. `vmOnly`

```
modifier vmOnly() {
    require(vm, "VM only");
    _;
}
```

作用：

-   只允许在 ReactVM 执行
    
-   常用于核心逻辑函数
    

最典型的就是：

```
function react(LogRecord calldata log) external vmOnly
```

* * *

# 订阅机制是怎么工作的

Reactive Contract 的前提是：**先订阅，再响应。**

* * *

## 1\. 核心接口：`ISubscriptionService`

主要有两个函数：

-   `subscribe(...)`
    
-   `unsubscribe(...)`
    

订阅时要指定：

-   `chain_id`：监听哪条链
    
-   `_contract`：监听哪个合约
    
-   `topic_0`：监听哪种事件
    
-   `topic_1/topic_2/topic_3`：是否进一步过滤 indexed 参数
    

* * *

## 2\. 核心接口：`IReactive`

最重要的是两部分：

### `LogRecord`

表示捕获到的一条事件日志，包含：

-   哪条链
    
-   哪个合约
    
-   哪个事件
    
-   topic 值
    
-   data
    
-   block number
    
-   tx hash
    
-   log index
    

### `Callback`

这是 Reactive 发起跨链/目标链调用的关键事件：

```
event Callback(
    uint256 indexed chain_id,
    address indexed _contract,
    uint64 indexed gas_limit,
    bytes payload
);
```

可以理解成：

> ReactVM 对 RN 说：“请帮我到目标链调用这个合约，并带上这段 calldata。”

* * *

# 静态订阅怎么理解

静态订阅就是：  
**在构造函数中一次性注册好监听规则。**

例如：

-   监听某个合约的某个事件
    
-   监听某条链上的某类事件
    
-   同时监听多个来源
    

注意：

```
if (!vm) {
    service.subscribe(...);
}
```

这句非常关键，意思是：

-   只有在 RN 环境中才真正注册订阅
    
-   ReactVM 不负责订阅
    

* * *

## 订阅时常见通配符

-   `address(0)` → 任意合约
    
-   `uint256(0)` → 任意链
    
-   `REACTIVE_IGNORE` → 任意 topic 值
    

* * *

## 静态订阅能做什么

### 场景 1：监听某个合约的所有事件

### 场景 2：监听全网某种固定事件

例如全网所有 Uniswap V2 `Sync` 事件

### 场景 3：监听某个合约的某个特定事件

这是最常用的方式。

* * *

# 订阅机制的限制

Reactive 的订阅不是 SQL 查询，它只是 **事件匹配**。

所以你必须记住几个限制。

* * *

## 1\. 不能在订阅时写范围判断

不能写：

-   amount > 1000
    
-   price < threshold
    

这些条件只能在 `react()` 里做。

订阅层只能做“匹配”，不能做数学比较。

* * *

## 2\. 不能在一次订阅里写 OR

想同时监听 A 合约和 B 合约，必须写两次 `subscribe()`。

* * *

## 3\. 不能监听“所有链上的所有事件”

这会造成灾难级资源消耗，系统禁止。

* * *

## 4\. 不能监听“某条链上的所有事件”

即使只限定一条链，也不行。  
你至少得给出更具体的目标：

-   某个合约
    
-   或某个事件
    

* * *

## 5\. 重复订阅 technically 可以，但会重复收费

同一个事件若被你重复订阅：

-   `react()` 会被触发多次
    
-   费用也会被扣多次
    

所以开发者必须自己避免重复订阅。

* * *

# 为什么取消订阅很贵

原因很简单：

-   `subscribe()` 更像是把规则追加到存储列表末尾
    
-   `unsubscribe()` 则需要先搜索，再删除，再填坑，再改长度
    

在 EVM 中：

-   读存储贵
    
-   改存储更贵
    

所以取消订阅通常比订阅更费 Gas。

* * *

# 动态订阅是什么

动态订阅不是在构造函数里全部写死，而是：

> 根据已经发生的事件，再新增或删除订阅。

但注意：

-   ReactVM 本身不能直接调用系统合约做订阅
    
-   真正的订阅动作仍然要在 RN 里完成
    

所以动态订阅要绕一圈：

1.  用户或某合约在源链发出“订阅请求事件”
    
2.  RN 捕获并转发到 ReactVM
    
3.  ReactVM 在 `react()` 中解析请求
    
4.  ReactVM 发出 `Callback`
    
5.  RN 收到 Callback 后调用本合约的 `subscribe(...)`
    
6.  合约再通过 `service.subscribe(...)` 完成新增订阅
    

所以动态订阅本质上是：

> **事件驱动的订阅管理**

* * *

# 动态订阅为什么要双重验证

当 RN 收到来自 Callback 的请求，要调用本合约里的 `subscribe()` / `unsubscribe()` 时，必须做安全校验。

至少两层：

## 1\. 验证直接调用者是系统服务合约

防止外部用户冒充 callback。

## 2\. 验证传入的 `rvm_id` 是正确的 owner / 合法来源

因为别的 ReactVM 也能发 callback，不能谁发都信。

一句话：

> **不是所有 callback 都可信，必须验证是谁发起的。**

# Uniswap V2 的核心原理

Reactive 止损单案例建立在 Uniswap V2 之上，所以必须理解池子和事件。

* * *

## 1\. Uniswap V2 不是订单簿，而是流动性池

传统交易所靠买单卖单撮合。  
Uniswap V2 靠一个池子完成交易。

池子里始终有两种代币，例如：

-   ETH
    
-   USDT
    

用户不是和另一个人交易，而是和池子交易。

* * *

## 2\. 恒定乘积公式

核心公式：

```
x * y = k
```

其中：

-   `x`：token0 储备
    
-   `y`：token1 储备
    
-   `k`：不变量
    

交易后必须保持乘积关系不被破坏（考虑手续费后）。

* * *

## 3\. 价格是储备金比值

Uniswap V2 没有单独存价格，价格是算出来的：

```
Price(token0) = reserve1 / reserve0
Price(token1) = reserve0 / reserve1
```

所以只要拿到最新储备金，就能推导价格。

* * *

# Uniswap V2 的两个关键事件

* * *

## 1\. `Swap`

表示发生了一笔兑换，记录：

-   谁调了 swap
    
-   输入多少
    
-   输出多少
    
-   发送给谁
    

这个事件更偏向“交易过程”。

适合做：

-   跟单
    
-   观察大户行为
    
-   套利分析
    
-   判断买卖方向
    

* * *

## 2\. `Sync`

表示储备金同步后的最终结果：

```
event Sync(uint112 reserve0, uint112 reserve1);
```

它记录的是：

-   当前池子的最新 reserve0
    
-   当前池子的最新 reserve1
    

这个事件更偏向“最终状态”。

适合做：

-   实时价格计算
    
-   止损止盈
    
-   流动性风险监控
    
-   跨链价格同步
    

* * *

## 为什么止损单更适合监听 `Sync`

因为你关心的是：

> 当前池子的真实储备比值有没有跌破阈值

而 `Sync` 给的正是交易后的最终储备。

* * *

# Reactive Network 如何结合 Uniswap V2

可以把流程记成三步：

## 1\. 订阅

监听某个 Uniswap Pair 的 `Sync` 或 `Swap`

## 2\. 过滤

在 `react()` 里解析事件数据，判断条件是否满足

## 3\. 执行

若满足条件，就发出 `Callback`，在目标链调用目标合约

# 止损单案例：UniswapDemoStopOrderReactive

今天最核心的案例就是这个。

它的目标是：

> 监控某个 Uniswap V2 流动性池，一旦价格跌破阈值，就自动在目标链执行止损。

* * *

## 1\. 核心业务模型

可以记成一句话：

> **Sync 事件 → 计算价格 → 若跌破阈值 → 触发 stop\_order.stop()**

* * *

## 2\. 合约里有哪些重要状态

### 生命周期状态

-   `triggered`：是否已经发出止损指令
    
-   `done`：整个流程是否彻底完成
    

### 业务参数

-   `pair`：监控哪个交易对
    
-   `stop_order`：真正执行止损操作的目标合约
    
-   `client`：这个订单属于谁
    
-   `token0`：监控价格方向
    
-   `coefficient`：放大系数，解决精度问题
    
-   `threshold`：价格阈值
    

* * *

## 3\. 为什么 `triggered` 和 `done` 很重要

它们本质上是在做状态机控制：

-   `triggered == false`：还没触发过，允许触发
    
-   `triggered == true`：已经发起过止损，防止重复下单
    
-   `done == true`：整个流程结束，后续不再处理任何事件
    

这是防重复执行的关键。

* * *

# 构造函数在做什么

构造函数做两件事：

## 1\. 初始化业务状态

把：

-   pair
    
-   stop\_order
    
-   client
    
-   token0
    
-   coefficient
    
-   threshold
    

全部存进状态变量。

## 2\. 在 RN 中注册订阅

只在 `!vm` 时执行：

-   订阅 Uniswap Pair 的 `Sync` 事件
    
-   订阅 stop\_order 合约发出的“停止完成”事件
    

第一条订阅用来发现价格变化。  
第二条订阅用来接收“执行完成确认”。

* * *

# `react()` 函数怎么理解

这个函数是整个响应式合约的核心大脑。

它有两个分支。

* * *

## 分支 1：收到 stop\_order 的确认事件

如果 `log._contract == stop_order`，说明这不是价格变动，而是执行反馈。

此时合约要验证：

-   已经触发过 `triggered == true`
    
-   `topic_0` 是目标完成事件
    
-   `topic_1` 是当前 pair
    
-   `topic_2` 是当前 client
    

这些检查通过后：

-   说明这条确认信号是给当前订单的
    
-   把 `done = true`
    
-   `emit Done()`
    

作用就是：

> 止损流程正式闭环结束

* * *

## 分支 2：收到 Uniswap Pair 的 `Sync` 事件

如果不是 stop\_order 发来的，那它就是价格更新。

步骤如下：

### 第一步：解码储备金

把 `log.data` 用 `abi.decode` 解成：

-   `reserve0`
    
-   `reserve1`
    

### 第二步：判断价格是否低于阈值

调用 `below_threshold(sync)`

并且还要满足：

-   `!triggered`
    

说明之前没发过止损指令。

### 第三步：组装回调 payload

用 `abi.encodeWithSignature(...)` 构造对目标止损合约的函数调用数据。

### 第四步：发出 Callback

```
emit Callback(...)
```

这不是普通事件，而是让 Reactive Network 去目标链发真实交易。

### 第五步：锁定状态

```
triggered = true;
```

防止后面又来一堆 `Sync` 导致重复止损。

* * *

# `below_threshold()` 在干什么

它负责根据储备金比值计算价格。

如果监控 `token0`：

```
price = reserve1 * coefficient / reserve0
```

如果监控 `token1`：

```
price = reserve0 * coefficient / reserve1
```

最后判断：

```
price <= threshold
```

* * *

## 为什么需要 `coefficient`

因为 Solidity 不支持浮点数。  
直接算 `reserve1 / reserve0` 很可能精度丢失。

所以先乘一个大系数，比如 `10^18`，把价格放大成整数，再比较阈值。

这也是 Uniswap 和 DeFi 里非常常见的整数化处理方式。

* * *

# 这个止损单案例的完整流程

把整个案例串起来，可以写成下面这条链：

1.  部署响应式合约
    
2.  构造函数初始化参数
    
3.  在 RN 中订阅：
    
    -   Pair 的 `Sync`
        
    -   stop\_order 的完成事件
        
4.  Uniswap 池子发生变化
    
5.  触发 `Sync`
    
6.  RN 捕获日志并转发给 ReactVM
    
7.  `react()` 解析 `reserve0/reserve1`
    
8.  `below_threshold()` 判断是否跌破阈值
    
9.  若跌破且未触发过：
    
    -   编码 stop 调用
        
    -   `emit Callback`
        
    -   `triggered = true`
        
10.  Reactive Network 的 Relayer 抓到 Callback
     
11.  在目标链执行 `stop_order.stop(...)`
     
12.  stop\_order 执行完成后发事件
     
13.  RN 再次捕获此事件并转发给 ReactVM
     
14.  `react()` 进入确认分支
     
15.  校验 pair/client/topic
     
16.  `done = true`
     
17.  止损单生命周期结束
     

* * *

# 这个案例里最重要的设计思想

今天这个案例最值得学的不是代码本身，而是背后的架构思维。

* * *

## 1\. 事件驱动，而不是轮询驱动

不是写个脚本一直去问：

-   价格变了吗？
    
-   到阈值了吗？
    

而是：

-   只要 `Sync` 一发生，系统立刻通知我
    

* * *

## 2\. 逻辑在 ReactVM，执行在目标链

ReactVM 做的是“判断”：

-   是否应该止损
    

目标链做的是“执行”：

-   真正调用 stop 函数
    

* * *

## 3\. 状态机控制生命周期

通过：

-   `triggered`
    
-   `done`
    

实现：

-   防重复触发
    
-   防重复完成
    
-   完成后彻底封口
    

* * *

## 4\. 通过事件确认闭环完成

不是你发了 Callback 就算完事。  
还要等目标合约发回“完成信号”。

也就是：

> **触发 ≠ 完成，必须等确认事件。**

# 今天这部分最容易考/最容易混的点

* * *

## 1\. `Swap` 和 `Sync` 的区别

-   `Swap`：交易过程
    
-   `Sync`：最终储备状态
    

止损场景更常用 `Sync`。

* * *

## 2\. RN 和 ReactVM 的区别

-   RN：管理订阅、接收用户交易、执行回调
    
-   ReactVM：处理事件逻辑、运行 `react()`
    

* * *

## 3\. 为什么 `subscribe()` 要放在 `if (!vm)`

因为订阅系统合约只存在于 RN，不在 ReactVM。

* * *

## 4\. 为什么要有 `vmOnly`

防止 `react()` 跑到 RN 上，造成逻辑混乱和高成本。

* * *

## 5\. `Callback` 不是普通日志

它是 Reactive Network 的“执行指令”。

* * *

## 6\. 为什么要先 `triggered = true` 再等确认

因为一个价格下跌时可能连续出现多个 `Sync` 事件，必须先锁住，防止重复发起止损。

## 核心范式：从被动到主动

传统的 EVM 智能合约是**被动**的，而 Reactive 实现了**自治**。

| 特性 | 传统智能合约 (Imperative) | 反应式智能合约 (Reactive/RSC) |
| --- | --- | --- |
| 触发机制 | 依赖外部调用 (EOA / Bot) | 监听链上事件 (Events) 自动触发 |
| 控制流 | 外部驱动 | 控制反转 (IoC)，合约自主响应 |
| 中心化程度 | 需托管私钥给 Keeper/Server | 逻辑上链，无需私钥托管，去中心化 |
| 跨链能力 | 需依赖第三方桥或预言机 | 原生支持多链事件订阅与回调 |

## 工作原理：Events + Callbacks “神经与肌肉”

Reactive 的核心在于将 **EVM 事件日志** 转化为 **自治执行流**。

### 1\. 事件 (Events) —— 系统的“感官”

-   **传统角色：** 仅作为不可修改的日志（Logs），供前端或 Indexer 查询。
    
-   **Reactive 角色：** 成为**触发源**。ReactVM 订阅特定日志（`chainId` + `address` + `topic0`），一旦目标链 `emit` 事件，Reactive Network 立即捕获并推送到执行层。
    

### 2\. 执行与回调 (Callbacks) —— 系统的“行动”

1.  **订阅 (Subscription)：** RSC 部署时声明要监听的事件签名。
    
2.  **检测 (Detection)：** Reactive Network 中继并解码日志数据。
    
3.  **评估 (Evaluation)：** 在 **ReactVM** 隔离环境中运行 Solidity 逻辑，判断是否满足触发条件。
    
4.  **回调 (Callback)：** 条件成立时，自动向目标链发起交易。
    

## 生态案例：十大应用场景概括

Reactive Network 的本质是：**把“人盯盘 + 手动操作”变成“链上自动程序”**。

1.  **DeFi 自动化：** 自动做市 (AMM) 调仓、复利收益 (Yield Compounding)、智能清算。
    
2.  **跨链套利：** 监控多链价差，自动执行对冲或搬砖。
    
3.  **NFT 工具：** 地板价监控、自动出价/扫货。
    
4.  **DAO 治理：** 监听投票结果，自动执行链上提案。
    
5.  **安全预警：** 大额异动、异常调用自动上链存证或触发阻断逻辑。
    
6.  **Oracle 联动：** 典型案例如 **Chainlink 价格集成**，监听价格更新事件（`AnswerUpdated`）实现止损/再平衡，无需外部 Keeper。
    

## **典型集成架构：Oracle 反应式闭环**

以 **Chainlink Aggregator** 为例：

1.  **Subscription:** RSC 监听 `topic0 == keccak256("AnswerUpdated(int256,uint256,uint256)")`。
    
2.  **Filter Logic:**$$If \\ |Price\_{current} - Price\_{last}| / Price\_{last} > Threshold\_{tolerance}$$
    
3.  **Action:** 调用 `targetChain.liquidate(user_address)`。 • **优势：** 消除 Keeper 节点，将“价格轮询”转变为“事件推送”，大幅降低运维开销。
    

## 开发实践与风险控制

### 1\. 关键技术问答

-   **关于公开性：** 合约字节码在链上可见，策略（如高频套利）存在被反编译风险，不建议存放需极度保密的短线交易策略。
    
-   **执行延迟：** 整个闭环（触发-传输-执行）约有 **10 秒** 延迟，适用于非毫秒级竞速场景。
    
-   **安全性：** AI 代理只需掌握“触发钱包”私钥而非“主资金池”私钥，大幅缩小风险半径。
    
-   **失败处理：** 跨链失败通常源于 Gas 不足。建议在合约中预设**失败回调逻辑**，并利用 `Reactscan` 排查。
    

### 2\. 开发三部曲

1.  **环境：** 使用 **Lasna Testnet**，参考 `Reactive Library` 的标准接口。
    
2.  **订阅：** 精确构造 `topics` 编码。
    
3.  **逻辑：** 在 `react()` 函数中实现条件判断，优化 Gas 消耗。
    

## 总结

Reactive Network 不是单纯的公链，而是**多链自动化中间层**。它通过 `Events + Callbacks` 构建了区块链的“自主意识”。

个人认为 Reactive Network 将智能合约从“死代码”变成了“自治代理”，并且对于致力于 Ethereum 基础设施研究的开发者来说，掌握这种**异步事件模型**是构建下一代去中心化应用的必经之路。
