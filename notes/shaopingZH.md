# 📝 笔记：订阅生命周期管理

在响应式编程中，你创建的每一条流，本质上都是一个**资源连接**（比如网络连接、数据库监听器、定时器等）。如果这些连接在不需要的时候没有被断开，它们就会像“僵尸”一样驻留在内存里，这就是最常见的**内存泄漏**来源。

### 1\. 为什么“订阅”需要管理？

当你调用 .subscribe() 时，框架会在内部建立一个监听管道。

-   如果你的页面销毁了，但管道还在（比如它还在等待网络响应），那么这个管道会持有该页面的引用，导致页面对象无法被垃圾回收（GC）。
    
-   这就是为什么在 Android、iOS 或前端框架（如 React/Vue）中，响应式流一定要挂载在组件的生命周期上。
    

### 2\. 核心工具：Disposable / Subscription

每个 subscribe() 方法在调用后，通常会返回一个对象（在 RxJava 中叫 Disposable，在 Kotlin 协程/Flow 中叫 Job，在 RxJS 中叫 Subscription）。

这个对象只有一个核心使命：**dispose() 或 cancel()**。

-   调用它，意味着告诉生产者：“我不玩了，后续的数据别再发给我了，赶紧把资源释放掉。”
    

### 3\. 最佳实践：如何优雅地“断开”

千万不要手动去记每一个 Disposable 变量，那会累死人，也容易漏。工业级的做法是使用\*\*“容器”\*\*。

A. 统一容器 (CompositeDisposable / CompositeSubscription)

创建一个容器，把所有该页面的订阅都扔进去。

code JavaScript

```
// 页面初始化
const disposables = new CompositeDisposable();

// 每次订阅时，把返回的 handle 加进容器
disposables.add(
  api.getData().subscribe(...)
);

// 页面销毁时 (onDestroy / componentWillUnmount)
disposables.clear(); // 或者 dispose()，一次性干掉所有连接
```

B. 自动解绑机制 (Lifecycle Binding)

在现代框架中，你甚至不需要手动调 clear()：

-   **Android (RxLifecycle / LifeCycle)**：框架可以直接监听 Activity/Fragment 的生命周期，自动在 onStop 或 onDestroy 时切断所有流。
    
-   **Kotlin (viewModelScope)**：在协程中，使用 viewModelScope 启动的流，会自动跟随 ViewModel 的销毁而自动取消，不需要写任何 dispose 代码。
    

### 4\. 避坑指南

-   **不要忽略 subscribe 的返回值**：写代码时，如果 IDE 提示你 Result of subscribe is ignored，请一定要重视。这通常意味着你创建了一个“失控”的资源。
    
-   **流的“自杀”行为**：有时候流处理完数据后会发送 onComplete 或 onError。如果流已经自然结束了，它会自动执行解绑，你不需要额外手动操作。只有那些**持续运行、永不结束的流**（如 UI 事件、Socket 监听），才必须通过 dispose() 手动干掉。
    
-   **千万别在异步回调里更新已销毁的 UI**：在解绑流的同时，如果你的后续逻辑包含 UI 更新，一定要确保断开连接后，代码不会再执行到更新界面的逻辑，否则会报“找不到 View”或者“Context 为空”的异常。

# 📝 笔记：时间轴控制

## 1\. 核心定义：把“时间”视为流的一部分

在响应式编程中，数据流不仅是在“空间”上（数据项）移动，它们还在“时间”轴上分布。**时间轴操作符**允许你根据事件发生的频率、间距或延时来调整流的节奏。

-   **一句话理解**：你不仅在控制“发什么数据”，还在控制“什么时候发”。
    

* * *

## 2\. 核心场景：三种经典的“防抖与节流”

在 UI 开发中，最经典的问题就是：用户疯狂点击“提交”按钮，如何防止请求发出去 10 次？

### 🚀 1. Debounce (防抖)

-   **做法**：给事件设一个“冷静期”。如果事件触发后，在规定时间（例如 300ms）内没有新的事件再次触发，才发射数据。
    
-   **形象比喻**：你是一个电梯管理员。有人按了电梯，你等了 3 秒，没人再按，你才关门。如果有人在 3 秒内又按了一下，计时器重新从 0 开始。
    
-   **场景**：**搜索框输入联想**。用户还没输入完，不要每打一个字就去查后台。等他停下输入 300ms 后再发送请求。
    

### 🐢 2. Throttle / Sample (节流/采样)

-   **做法**：在一段时间内，只允许发射一次（或者是固定频率发射）。
    
-   **形象比喻**：单位时间内只处理一个请求。不管你在这段时间内触发了多少次，我只听第一次的，或者只听最后一次的。
    
-   **场景**：**滑动加载/滚动监听**。当用户滑动页面时，滚动事件一秒钟触发上百次。你需要用 sample 限制为 200ms 采样一次，避免疯狂触发“检查是否到底”的逻辑。
    

### ⏳ 3. Delay (延时)

-   **做法**：将整个流的发射时间整体向后平移。
    
-   **场景**：**优雅的 Loading 交互**。如果网络请求极快（10ms 就回了），UI 上的 Loading 转圈动画可能只闪一下，非常难看。用 delay 让响应延迟 300ms，保证用户能看到 Loading 的转圈效果。
    

* * *

## 3\. 三大操作符对比图解

| 操作符 | 行为逻辑 | 典型应用 |
| debounce | 只有当“安静”一段时间后才发送 | 搜索框防抖、表单实时校验 |
| throttleFirst | 发送第一个，忽略后续，直到一段时间结束 | 防连点（提交按钮） |
| sample | 固定时间周期内，只取最新一个 | 滚动位置监听、实时进度刷新 |

* * *

## 4\. 避坑指南 (Note to Self)

### 💣 陷阱 1：忘了指定 Schedulers

**警告**：大部分时间操作符（如 debounce, delay, timer）默认并不是在主线程执行的。  
**后果**：如果你在 debounce 之后紧接着操作 UI（比如 textView.setText()），而没有使用 observeOn(MainThread)，你的应用会因为线程问题直接报错！

### 💣 陷阱 2：switchMap 是防抖的神器

**进阶小技巧**：在处理搜索请求时，除了用 debounce，一定要结合 switchMap。

-   switchMap 的特性是：一旦上游又发了一个新请求，它会\*\*自动取消（Dispose）\*\*上一次正在进行中的网络请求。
    
-   这不仅解决了防抖，还解决了“后发出的请求比先发出的请求先返回导致数据错乱”的问题。
    

### 💣 陷阱 3：测试的噩梦

**场景**：你写了一个 delay(60000) (延迟一分钟) 的逻辑。  
**后果**：写单元测试时，你真的要等 60 秒吗？  
**对策**：响应式框架通常提供 **TestScheduler**。在测试代码中，你可以通过手动拨动“虚拟时钟”，瞬间触发 1 分钟后的逻辑，无需真的等待。

# 📝 笔记：多流组合

## 1\. 核心定义：从“单机”到“联机”

在实际业务中，你的数据很少是单一来源的。你可能需要同时监听“用户输入”、“网络状态”和“定时器”，并根据它们的综合状态来刷新 UI。  
多流组合操作符，就是用来处理**多个 Observable 之间的协同关系**的（比如：并行、依赖、竞速、合并）。

告别繁琐的 boolean flag1, flag2 和深层嵌套回调，用声明式的方式定义并发逻辑。

* * *

## 2\. 核心心智模型：鸡尾酒调酒师

想象你是一个高级调酒师，面前有多个管道（数据流）正在流出不同的液体：

-   **A 管道**：每隔几秒流出一滴伏特加。
    
-   **B 管道**：随时可能流出橙汁。
    
-   **C 管道**：流出冰块。
    

你的任务是使用不同的“混流机器（操作符）”，将它们按特定的规则混合成一杯杯完美的鸡尾酒。

* * *

## 3\. 最常用的三大组合“机器” (The Big Three)

这是面试和实战中最常见的三个混流操作符，搞懂它们就能应对 90% 的并发场景：

### 🔀 1. Merge (无序交替合并)

-   **核心逻辑**：**“先到先得” (OR 逻辑)**。
    
-   **行为**：将多个流的内容直接合并到一个流里。谁先发射数据，谁的数据就先进入下游。
    
-   **形象比喻**：高速公路上的两条车道合并成一条。不管你是左车道还是右车道，谁开得快谁就先过去。
    
-   **常见场景**：你有一个“下拉刷新”流和一个“点击加载”流，这两个流都会触发同一个“请求列表数据”的逻辑。用 merge 把它们合二为一。
    

### 🤐 2. Zip (严格拉链配对)

-   **核心逻辑**：**“齐头并进，缺一不可” (AND 逻辑)**。
    
-   **行为**：分别从 A 流和 B 流取**第 N 个**数据，把它们配对组合后再发给下游。如果 A 发了数据，B 还没发，**下游就会死等 B**。
    
-   **形象比喻**：组装汉堡。A 流出面包，B 流出肉饼。必须等一个面包和一块肉饼都到了，才能拼成一个汉堡。如果面包一直来，肉饼没来，面包只能在流水线上排队等着。
    
-   **常见场景**：进入一个页面，需要同时请求“用户信息接口”和“用户配置接口”。两者互相独立（并发请求），但必须**两个都返回成功**后，才能渲染页面。
    

### 🔄 3. CombineLatest (结合最新值)

-   **核心逻辑**：**“牵一发而动全身” (动态更新)**。
    
-   **行为**：只要 A 或 B 任何一个流发射了新数据，它就会立刻拉取另一个流的**最新缓存值**，把两者结合发给下游。
    
-   **形象比喻**：购物车总价计算。A 流是“商品单价”（可能变动），B 流是“购买数量”（用户不断点击+1）。无论单价变了，还是数量变了，总价都会立刻重新计算。
    
-   **常见场景**：极其适合做**表单校验**。监听“账号输入流”、“密码输入流”和“同意协议打勾流”，只要任意一个发生变化，就立刻结合三个流的最新值，判断“登录”按钮是否应该高亮。
    

* * *

## 4\. 经典代码示例 (以表单校验为例)

感受一下没有任何 if-else 和全局变量的优雅：

code JavaScript

```
// 假设有三个独立的 UI 交互流 (Hot Streams)
const usernameStream = inputUsername.textChanges();
const passwordStream = inputPassword.textChanges();
const checkboxStream = checkbox.checkedChanges();

// 魔法开始：将它们组合在一起
Observable.combineLatest(
    usernameStream,
    passwordStream,
    checkboxStream
).map(([username, password, isChecked]) => {
    // 任何一个流有动静，都会进入这里重新计算
    const isUserValid = username.length >= 4;
    const isPassValid = password.length >= 6;
    return isUserValid && isPassValid && isChecked; 
})
.subscribe(
    isFormValid => {
        // UI 渲染逻辑
        loginButton.setEnabled(isFormValid);
        loginButton.setAlpha(isFormValid ? 1.0 : 0.5);
    }
);
```

* * *

## 5\. 避坑指南 (Note to Self)

### 💣 致命陷阱 1：Zip 导致的内存泄漏

-   **场景**：A 流疯狂发射数据（1秒100个），B 流半天才发射一个数据（10秒1个）。
    
-   **后果**：因为 Zip 必须严格的一对一配对，A 流多出来的 99 个数据不能扔，只能缓存在内存里死等 B。时间一长，内存直接撑爆（OOM）。
    
-   **对策**：使用 Zip 时，确保两个流的数据产生速率大致相同，或者用于那种“只会发射一次”的网络请求流。
    

### 💣 致命陷阱 2：CombineLatest 的“启动死锁”

-   **场景**：你发现 CombineLatest 下游完全没有反应。
    
-   **原因**：CombineLatest 有一个前置条件：**所有参与的流，都必须至少发射过一次数据**，它才会开始第一次组合。如果 A 流发了，B 流永远保持沉默，那下游连一次数据都收不到。
    
-   **对策**：给那些可能很久才有数据的流，加上一个 .startWith(默认值) 操作符，给它一个初始状态，打破僵局。

# 📝 笔记：线程调度

## 1\. 核心定义：打破“默认同步”的幻觉

首先必须牢记一个残酷的真相：**响应式流默认是同步的！**  
如果你不显式地指定线程，你从哪里 subscribe()，整个流水线（包括网络请求、数据处理）就都在哪个线程里执行。如果你在 UI 线程里发起了网络请求流，恭喜你，你的 APP 直接卡死（ANR）。

**调度器（Schedulers）就是用来解决这个问题的。它是响应式编程中的线程池管家**，让你能够用极其优雅的声明式代码，随意切换代码执行的线程。

* * *

## 2\. 核心心智模型：跨车间流水线

想象一条极长的自动化产品流水线，它穿过了三个不同的车间：

1.  **采矿车间 (IO 线程)**：负责干脏活累活（网络请求、读写数据库）。
    
2.  **加工车间 (Computation 线程)**：负责高强度的精密计算（解析大量 JSON、图片滤镜计算）。
    
3.  **展示大厅 (Main/UI 线程)**：非常脆弱且尊贵，只能用来把成品摆上展台（更新 UI 界面），绝对不能在这里干重活。
    

我们的目标是：让材料在正确的车间里被处理。

* * *

## 3\. 灵魂拷问：subscribeOn vs observeOn

这是面试必考题。这两个操作符虽然长得像，但作用域完全不同：

### ⬆️ subscribeOn (改变上游/源头线程)

-   **作用**：决定这条流**从哪里开始发射数据**。也就是决定了“采矿车间”建在哪里。
    
-   **核心法则**：**“只有第一次有效，位置随便放”**。
    
    -   如果你在一条流里写了多个 subscribeOn，只有最靠近源头的那个（通常是第一个）会生效。后面的会被忽略。
        
    -   无论你把 subscribeOn 写在流水线的哪一行，它都会影响源头的线程。
        

### ⬇️ observeOn (改变下游/后续线程)

-   **作用**：决定**在这个操作符之后**的代码，在哪个线程执行。也就是“把传送带引向哪个车间”。
    
-   **核心法则**：**“可以调用无数次，每次都影响它下面的代码”**。
    
    -   你可以多次调用 observeOn，让数据在不同的线程之间反复横跳。

# 📝 笔记：背压机制 (Backpressure)

## 1\. 核心定义

**背压（Backpressure）是一种处理“生产者发送数据的速度，远远超过消费者处理数据的速度”**这一上下游速率不匹配问题的机制。

简单来说：它是消费者向生产者发送的一种“反馈信号”，告诉生产者：“我处理不过来了，请慢点发，或者按我的节奏发！”

* * *

## 2\. 核心心智模型：消防栓喝水

想象一下你要喝水：

-   **正常情况（无背压需求）**：你拿着一个水杯接饮水机的水，水流平缓，你喝得游刃有余。
    
-   **异常情况（需要背压）**：有人把**高压消防栓**塞进了你的嘴里，一秒钟喷出 100 升水，而你一秒钟只能咽下 1 口水。
    

**如果不做任何干预，后果就是：**

1.  你的嘴巴（内存/缓冲区）被撑爆。
    
2.  你整个人宕机（程序 Out Of Memory, 内存溢出 Crash）。
    

背压机制，就是给你一个阀门开关，让你自己决定放多少水出来。

* * *

## 3\. 常见的背压处理策略 (The Strategies)

当上游数据像洪峰一样涌来，而下游处理缓慢时，框架通常提供以下几种“泄洪”策略：

### ① 缓冲策略 (Buffer)

-   **做法**：搞一个大水池（队列），把处理不过来的数据先存起来，慢慢处理。
    
-   **缺点**：如果水一直猛灌，水池迟早会满，最终依然会导致 OOM（内存溢出）。只能应对**阵发性**的数据高峰。
    

### ② 丢弃策略 (Drop)

-   **做法**：既然处理不过来，那多出来的数据我直接扔掉不管了。
    
-   **场景**：适合**实时性要求高、但允许丢失部分数据**的场景（比如：股票实时价格看板、GPS 坐标持续上报）。
    

### ③ 保留最新策略 (Latest)

-   **做法**：和 Drop 类似，但我永远保留最新的一条数据。当下游空闲时，直接处理最新鲜的那条。
    
-   **场景**：UI 界面更新（比如下载进度条，如果一秒更新一万次没意义，用户只关心当前的最新进度）。
    

### ④ 响应式拉取 (Reactive Pull / Request)

-   **这是最核心的终极背压方案**。
    
-   下游消费者不再是被动挨打，而是主动向生产者声明：request(n)（我只能处理 n 个，你先给我发 n 个，等我处理完了再找你要）。
    
-   _类比：去餐厅吃饭，吃完一盘再叫服务员上一盘。_
    

* * *

## 4\. 框架中的具体体现 (以 RxJava 为例)

在早期（RxJava 1）中，所有东西都混在一起，经常报错。后来框架设计者为了区分，强制把 API 拆分了：

-   **Observable (不支持背压)**：  
    仅仅是一个普通的流。适用于数据量小、发生频率低的情况（如鼠标点击事件、单个网络请求）。如果上游狂发数据，下游直接死给你看。
    
-   **Flowable (支持背压)**：  
    专门为了背压设计的流。适用于海量数据、高频发送的情况（如读取大文件、数据库大量查询）。强制你必须选择一种背压策略（Buffer/Drop/Latest）。
    

* * *

## 5\. 避坑指南 (Note to Self)

-   **滥用背压的陷阱**：不要为了炫技什么都用背压（比如 Flowable）。处理一个用户的登录请求，或者监听一个按钮点击，完全不需要背压！**只有在数据量大于 10,000 条/秒 的持续流中，才需要考虑背压。**
    
-   **下游阻塞是万恶之源**：背压往往是因为下游在执行耗时操作（比如在流里写数据库、做复杂计算）。在考虑使用 Drop 策略丢弃数据前，先看看能不能优化下游的处理速度。
    
-   **报错识别**：如果你在日志里看到了大名鼎鼎的 MissingBackpressureException，不要慌，这就是上游发疯了，赶紧去把 Observable 换成有背压策略的 Flowable，或者加一个 .onBackpressureDrop() 操作符。

**map 与 flatMap 的区别**

在处理数据流时，我们经常需要对流中的元素进行转换。map 和 flatMap 都是用来做转换的，但它们的“维度”处理完全不同。

## 1\. Map (一对一转换)

-   **功能**：对流中的每一个元素进行同步的、简单的变换。
    
-   **输入输出**：输入一个数据，经过函数处理，输出一个**数据**。
    
-   **形象比喻**：你有一筐苹果（数据流），map 是一台“削皮机”，你放进一个苹果，出来一个削好皮的苹果。**数量关系始终是一对一**。
    

**代码示例 (以数组为例)：**

code JavaScript

```
// [1, 2, 3] -> [2, 4, 6]
const numbers = [1, 2, 3];
const doubled = numbers.map(x => x * 2);
```

* * *

## 2\. FlatMap (一对多/扁平化转换)

-   **功能**：当你的转换逻辑本身又会产生一个新的“流”时，使用 flatMap。
    
-   **输入输出**：输入一个数据，转换成一个**子流 (Inner Stream)**，然后将所有子流的元素“平铺”汇聚成一个大的流。
    
-   **形象比喻**：你有一筐苹果，flatMap 是一台“榨汁机”。你放进一个苹果（输入），它会变出一堆果汁分子（一个新的流），最后所有苹果产生的果汁汇聚成一条长长的果汁流。
    

**核心痛点**：如果你用 map 处理异步逻辑（例如：输入ID -> 发起网络请求），map 会返回一个“流的流”（Observable<Observable<Result>>），你没法直接拿结果，必须用 flatMap 把内部的流“拉平”拿到结果。

* * *

## 3\. 对比总结表

| 特性 | map | flatMap |
| 转换对象 | 将元素转换为另一个值 | 将元素转换为另一个流 |
| 结果形式 | 转换后的值 | 多个流合并后的平铺结果 |
| 使用场景 | 数据简单变换 (如 x + 1) | 处理异步操作 (如 ID 转用户信息) |
| 维度 | 保持原有维度 | 降维（拉平嵌套的流） |

* * *

## 4\. 深度理解：为什么会有“嵌套流”？

这是响应式编程中最难的部分。假设你要根据用户 ID 查询用户信息：

-   **如果用 map**：  
    id -> fetchUser(id)  
    因为 fetchUser 是异步的，它返回的是一个 Observable<User>。  
    结果变成了：Observable<Observable<User>> (嵌套流)。你无法直接使用 User 对象。
    
-   **如果用 flatMap**：  
    它会自动执行“订阅内部流”并把结果“平铺”到外层的流中。  
    结果变成了：Observable<User> (直接拿到了用户对象)。
    

* * *

## 5\. 常见陷阱与建议

### 陷阱：flatMap 的并发性

flatMap 是**并发执行**的。如果你连续发出请求，flatMap 内部产生的子流会乱序（谁先返回谁先出来）。

-   **如果你需要保证顺序**：某些库（如 RxJava/Project Reactor）提供了 concatMap，它会等待上一个子流执行完，再执行下一个，从而保证顺序。
    

### 避坑指南：

-   **看到异步就选 flatMap**：当你发现你的转换函数里调用了另一个返回 Observable/Publisher/Promise 的方法时，几乎 99% 的情况你应该选 flatMap。
    
-   **看到同步计算就选 map**：如果转换不需要耗时，也不产生新的流，永远选 map，它性能更高，逻辑更简单。

# 📝 响应式编程 (Reactive Programming) 核心笔记

## 1\. 核心定义

响应式编程是一种基于\*\*数据流（Data Streams）**和**变化传播（Propagation of Change）\*\*的异步编程范式。

-   **一句话理解**：你不再是“请求-响应”模式，而是建立一个“流水线”，当数据产生时，它会自动流过处理节点，并最终到达消费者。
    
-   **对比传统模式**：
    
    -   **传统（Pull）**：消费者主动去问生产者“有数据了吗？”（Iterator 模式）。
        
    -   **响应式（Push）**：生产者有数据了，主动“推送”给消费者（Observer 模式）。
        

* * *

## 2\. 核心四大支柱 (The Core Pillars)

大多数响应式框架（RxJava, Project Reactor, RxJS, Combine 等）都遵循这四个核心概念：

### ① Observable (被观察者 / 数据源)

-   数据的生产者。
    
-   负责产生事件（发射数据项、发射错误、发送完成信号）。
    
-   _类比：流水线的源头。_
    

### ② Observer (观察者 / 消费者)

-   数据的订阅者。
    
-   对来自 Observable 的数据做出反应。
    
-   _类比：流水线的终点/处理工位。_
    

### ③ Operators (操作符)

-   **这是响应式编程的灵魂**。
    
-   它允许你像操作集合一样处理异步流（过滤、转换、合并、延时）。
    
-   常见例子：map (转换), filter (过滤), flatMap (平铺), zip (合并)。
    
-   _类比：流水线上的加工机器（切片、涂装、组装）。_
    

### ④ Schedulers (调度器)

-   控制代码在哪个线程执行。
    
-   将复杂的线程切换（Context Switching）抽象化。
    
-   _类比：流水线工人的排班表（决定是在哪个车间干活）。_
    

* * *

## 3\. 核心心智模型：流水线 (The Assembly Line)

想象一个自动化的流水线：

1.  **输入**：原料（数据）从源头进入。
    
2.  **加工**：经过一系列机器（Operators：先洗涤、再切片、再包装）。
    
3.  **输出**：最终产品到达客户手中（Observer）。
    

**关键点**：只要源头不断提供原料，整个流水线就会一直运转，不需要你手动去拉动每一个环节。

* * *

## 4\. 为什么要用？ (Pros & Cons)

### ✅ 优势

-   **代码声明式**：代码逻辑清晰，告诉程序“做什么”，而不是“怎么做”。
    
-   **异步解耦**：极大地简化了异步代码（告别“回调地狱”）。
    
-   **流式处理**：天然适合处理高频、连续的数据（如 UI 点击、传感器数据、实时推送）。
    
-   **组合性强**：操作符可以随意组合，逻辑复用性高。
    

### ⚠️ 挑战（学习曲线）

-   **思维转换困难**：从“命令式”转变为“声明式”思维需要时间。
    
-   **调试难度**：由于是异步堆栈，报错时很难定位（Stack Trace 往往很长且难以理解）。
    
-   **操作符黑洞**：API 过于庞大，容易迷失在数百个操作符中。
    

* * *

## 5\. 快速上手示例 (伪代码)

// 假设我们要处理一个用户点击事件流
clickStream
  .filter(event => event.isValid)      // 1. 过滤：只处理有效点击
  .map(event => event.value)           // 2. 转换：提取数据值

## 6\. 避坑指南 (Note to Self)

-   **冷流 vs 热流 (Cold vs Hot)**：
    
    -   **冷流**：不被订阅就不开始工作（通常是数据流）。
        
    -   **热流**：创建时就开始工作，不管有没有人订阅（通常是事件流，如鼠标点击）。
        
-   **内存泄漏**：在 Android 或 UI 开发中，**一定要记得取消订阅（dispose/unsubscribe）**，否则会造成严重的内存泄漏。
    
-   **不要过度使用**：如果是简单的逻辑，直接用普通函数即可，不要为了“响应式”而“响应式”。
    

* * *

_💡 笔记建议：学习响应式最好的方式不是死记 API，而是去理解 Observable 和 Observer 之间的契约关系，以及 map 和 flatMap 的区别。_
