## 内容

目前的系统在以下几个方面存在一些技术限制和问题：

**1\. 事件订阅机制的局限性 (Subscription Limitations)**

-   **仅支持精确匹配**：目前的订阅系统只支持对事件的精准匹配，不支持使用 `<`（小于）、`>`（大于）、数值范围过滤或按位过滤等高级条件。
    
-   **不支持复杂条件组合**：每个订阅只能定义一组固定的匹配条件，不支持使用“或（OR）”这样的逻辑运算。如果需要监听多种情况，开发者只能通过多次调用 `subscribe()` 来变通，但这会导致订阅数量增加以及消耗更多的 Gas 费。
    
-   **禁止全局订阅**：系统出于性能和安全考虑，不允许开发者设置全局订阅，例如不能订阅“所有链的事件”、“所有合约的事件”或“某条特定链上的所有事件”。
    

**2\. ReactVM 隔离环境的限制 (ReactVM Limitations)**

-   **无法在处理逻辑时直接管理订阅**：在 ReactVM 隔离环境中运行（例如正在 `react()` 函数中处理事件）时，直接调用 `subscribe()` 或 `unsubscribe()` 是完全无效的。必须通过向外触发 Callback 回调事件，让 Reactive 主网状态介入才能实现动态订阅的修改，这增加了开发的复杂度。
    
-   **跨合约交互隔离**：ReactVM 是与部署者地址绑定的私有隔离环境。在这个环境中运行的合约，**不能**与其他用户部署的合约进行直接交互。
    

**3\. Gas 与执行限制 (Gas and Callback Limits)**

-   **执行上限**：在 ReactVM 中执行单笔交易的最高 Gas 限制为 900,000 个单位。
    
-   **回调下限**：跨链回调存在最低 100,000 Gas 的强制要求。任何低于此数值的回调请求都会被系统直接忽略，因为系统需要保留足够的 Gas 进行内部审计和计算。
    

**4\. 经济与服务中断风险 (Economy Risks)**

-   系统采用“先执行后记账”的模式，如果开发者的合约余额不足以支付产生的费用，合约会被自动列入黑名单（Blocklisted）并变为非活跃状态（inactive）。一旦发生这种情况，合约将无法执行任何操作或跨链回调，除非充值清算债务，这要求开发者必须密切关注资金余额以防止服务意外中断。

## 内容

### Reactive 合约的双状态执行机制与模式解析

Reactive 合约（Reactive Contracts）存在于一个独特的\*\*双状态环境（Dual-State Environment）\*\*中，同一份字节码会同时部署在两个独立的环境里，并维护各自独立的状态。基于这种架构，Reactive 合约具有两种截然不同的执行模式：

**1\. ReactVM 模式 (**`vmOnly`**)**

-   **运行环境**：在私有且隔离的 **ReactVM** 中执行。
    
-   **触发方式**：它是**自动触发**的。当合约订阅的区块链事件在源链上发生时，ReactVM 会捕获该事件日志并自动唤醒合约执行逻辑。
    
-   **状态更新**：在此模式下更新的是 ReactVM 内部的状态。
    
-   **核心用途**：绝大多数的**事件处理和自动化逻辑**都在这个模式下运行。例如，接收到特定事件后执行计算、判断条件，并最终决定是否向目标链发送跨链回调（Callback）。
    
-   **代码限制**：在 ReactVM 中调用系统合约（如 `subscribe()` 或 `unsubscribe()`）是无效的，因为系统合约在该环境中不存在。开发者在代码中使用 **vmOnly** 修饰符来确保某些核心自动化函数只能在 ReactVM 中被调用。
    

**2\. Reactive Network 模式 (**`rnOnly`**)**

-   **运行环境**：在 **Reactive Network (RNK) 的公共主网状态**中执行。
    
-   **触发方式**：它是**被动触发**的。只有当外部拥有账户（EOA，即真实用户或开发者）直接发起交易调用合约函数，或者通过回调交易触发时，才会在该模式下执行。
    
-   **状态更新**：在此模式下更新的是 Reactive Network 主网上的公共状态。
    
-   **核心用途**：通常用于执行**管理性操作**。例如，合约所有者手动调用 `pause()` 暂停合约、动态管理事件的订阅与取消订阅、或者为合约和回调代理充值、结算费用等。
    
-   **代码限制**：开发者在代码中使用 **rnOnly** 修饰符来限制这些管理函数只能在公共网络中执行。
    

**系统如何识别执行模式？** 当你继承了 Reactive 官方提供的 `AbstractReactive` 基础合约后，系统会通过内置的 **detectVm()** 函数自动检测当前的运行环境。它主要通过检查系统合约（System Contract）在当前上下文中是否已部署来判断：如果系统合约不存在，则说明合约正在 ReactVM 隔离环境中运行（即 `vmOnly`）；反之则是在 Reactive Network 主网中运行（即 `rnOnly`）

## 内容

在 Reactive Network 中，可以通过系统合约（System Contract）内置的 **CRON 机制**来实现定时或周期性操作，完全不需要依赖任何外部的定时轮询机器人。

以下是使用该机制的核心步骤和原理：

**1\. CRON 机制的工作原理**

与传统的智能合约只能被动响应用户交易不同，Reactive Network 的系统合约会在固定的区块间隔自动发出 `Cron` 事件。授权的验证节点会调用 `cron()` 函数，根据当前区块号能否被特定数值整除，来触发不同频率的定时事件。这相当于给智能合约提供了一个链上的“内置时钟”。

**2\. 如何订阅和使用**

要实现定时操作，你的 Reactive 合约只需要像监听普通链上事件一样，**订阅这些特定的 Cron 事件**。每次事件触发时，它都会携带一个参数 `number`，代表当前的区块号。

系统目前提供 5 种不同频率的定时事件，你可以在合约的订阅逻辑中使用对应的 Topic 0 来监听它们：

-   **Cron1（每区块触发一次，约 7 秒）**：Topic 0 `0xf02d6ea5c22a71cffe930a4523fcb4f129be6c804db50e4202fb4e0b07ccb514`
    
-   **Cron10（每 10 个区块触发，约 1 分钟）**：Topic 0 `0x04463f7c1651e6b9774d7f85c85bb94654e3c46ca79b0c16fb16d4183307b687`
    
-   **Cron100（每 100 个区块触发，约 12 分钟）**：Topic 0 `0xb49937fb8970e19fd46d48f7e3fb00d659deac0347f79cd7cb542f0fc1503c70`
    
-   **Cron1000（每 1000 个区块触发，约 2 小时）**：Topic 0 `0xe20b31294d84c3661ddc8f423abb9c70310d0cf172aa2714ead78029b325e3f4`
    
-   **Cron10000（每 10,000 个区块触发，约 28 小时）**：Topic 0 `0xd214e1d84db704ed42d37f538ea9bf71e44ba28bc1cc088b2f5deca654677a56`
    

**3\. 典型的应用场景**

结合 CRON 机制，开发者可以构建出丰富的纯链上自动化任务：

-   **借贷防清算保护（Health Checks）**：例如官方提供的 Aave 防清算演示中，Reactive 合约就是通过订阅定期的 CRON 事件，每隔一段时间自动检查用户在 Aave 上的头寸健康因子（Health Factor）。一旦发现跌破阈值，合约就会触发跨链回调，自动追加抵押物或偿还债务以防止清算。
    
-   **定期发放奖励（Reward Distributions）**：无需项目方手动操作或设置链下脚本，即可根据固定的区块间隔按时向用户分配收益。
    
-   **定期数据更新（Data Updates）**：定期触发信息同步，如抓取最新的预言机价格或定期执行跨链资金池的再平衡逻辑。

CRON 定时器在 DeFi 借贷清算保护中如何应用？

在 DeFi 借贷（例如 Aave）的清算保护中，Reactive Network 内置的 CRON 定时器主要充当**链上自动触发时钟**的角色，用于彻底替代传统的链下监控机器人。

结合官方提供的 Aave 防清算演示（Aave Liquidation Protection Demo），CRON 定时器的具体应用流程如下：

1.  **订阅定时事件**：用户部署专属的个人响应式合约（Reactive Contract），并让该合约订阅特定频率的 CRON 事件（例如每 10 个区块或每 100 个区块触发一次的事件）。
    
2.  **定期健康度检查（Health Checks）**：每当订阅的 CRON 事件在底层网络固定触发时，响应式合约就会被自动唤醒，并针对用户在借贷平台上的借贷头寸执行“健康因子（Health Factor）”检查。
    
3.  **条件判断与跨链回调**：如果合约在检查中发现用户的健康因子跌破了用户自定义的安全阈值，它会立即触发一个跨链回调（Callback）指令到目标链。
    
4.  **自动执行清算保护**：目标链上的回调合约接收到指令后，会自动计算出将健康因子恢复到安全线所需的金额，并直接与 Aave 交互来执行保护措施——通常是**自动追加存入抵押物**，或者**自动偿还部分债务**。
    

**应用优势：** 通过 CRON 定时器，这种基于时间的自动化清算保护机制实现了完全的链上闭环。用户**无需时刻手动监控市场行情，也无需依赖或运行任何外部的链下机器人**即可保障头寸安全。此外，每个用户都可以部署自己专属的隔离保护设置，确保资金管理的绝对控制权。

## 内容

响应式合约（Reactive Contracts, RCs）与传统智能合约的主要区别在于**执行机制（主动与被动）控制反转（Inversion of Control）**：

-   **传统智能合约是“被动的”**：它们遵循直接控制模型，只有在接收到外部拥有账户（EOA，如真实用户或链下机器人）直接发起交易调用时，才会执行相应的逻辑。
    
-   **响应式合约是“主动响应的”**：它们具有自主性，能够持续不断地监控区块链上发生的特定事件。一旦检测到这些预定义的事件，响应式合约就会自动触发并执行相关的操作，整个过程无需任何外部用户或机器人的干预。
    

这种核心差异引入了“控制反转（Inversion of Control, IoC）”的范式。在传统架构中，必须依赖链下的中心化机器人来监听数据并用私钥签名发起交易；而响应式合约将控制权交给了合约本身，让合约能够基于链上事件自主决定何时执行，从而实现了一个更加动态、自动化且完全去中心化的系统。

传统的智能合约是“被动”的，必须依靠外部拥有账户（EOA，通常是链下的中心化机器人）来轮询数据并签名发送交易才能执行逻辑。而响应式合约（Reactive Contracts, RCs）是“主动”的，它们能够全天候自主监控区块链事件，并在满足条件时自动执行指令。

具体实现无需机器人的自动化，主要依靠以下四个核心机制：

**1\. 链上事件自动订阅与监听 (Subscriptions)** 开发者可以通过调用系统合约，直接让响应式合约订阅特定链上的事件（通过指定源链 ID、合约地址和事件 Topics）。这种网络原生支持的订阅机制替代了过去需要由链下机器人不断轮询（Polling）和抓取区块链数据的中心化组件。

**2\. ReactVM 隔离环境与自动唤醒 (ReactVM Processing)** 响应式合约运行在专属的私有隔离执行环境 **ReactVM** 中。当订阅的链上事件发生时，Reactive Network 的底层节点会自动捕获事件日志，并将其直接推送到 ReactVM 内，自动唤醒并调用合约的 `react()` 函数执行 Solidity 逻辑。这一唤醒过程是网络共识机制的一部分，完全不需要任何外部机器人发送触发交易。

**3\. 去中心化跨链回调 (Cross-Chain Callbacks)** 当合约在 `react()` 函数中计算完毕并决定采取行动时（例如发现价格跌破预设的止损线），合约不需要调用外部的跨链桥或机器人程序，只需在链上触发（Emit）一个 `Callback` 事件。该事件内部包含了目标链 ID、目标合约地址和编码后的函数数据（Payload）。Reactive Network 监测到此回调事件后，会自动将该交易提交并广播到目标链上执行。

**4\. 替代定时机器人的内置时钟 (CRON Functionality)** 对于许多需要基于时间（而非特定用户交互）执行的自动化任务（例如定期分发奖励、DeFi 借贷健康度检查等），系统合约提供了原生的 CRON 机制。它会在固定的区块间隔（例如每区块、每 10 个区块、每 1000 个区块等）自动发出 Cron 事件。合约只需订阅这些事件，即可获得一个“内置时钟”，从而实现定时计划任务，彻底消除了对链下定时机器人的依赖。

通过将“事件监听 -> ReactVM 自动计算 -> 跨链回调执行”这一整套流程全部封装在底层网络协议中，开发者只需编写纯 Solidity 代码，只要你能用条件语句表达逻辑，Reactive Network 就能在链上自动完成跨链执行
