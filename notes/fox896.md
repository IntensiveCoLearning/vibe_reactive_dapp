4.1 订阅机制如何运作

本节课，我们将深入探讨订阅，这是响应式网络的一个关键特性，它允许响应式网络自动响应其他合约发出的事件。当这些事件发生时，订阅合约可以自动执行预定义的逻辑。

4.1.1 如何实现订阅

subscribe在响应式合约中，订阅是通过响应式网络系统合约中的方法设置的。通常，这是在构造函数中初始化订阅的过程，不过订阅也可以动态管理。我们将在本文末尾讨论动态订阅。

响应式合约还必须处理由于在响应式网络（具有系统合约）和其部署者的私有 ReactVM（其中不存在系统合约）上部署而导致的回滚。

3.1 ReactVM 和 Reactive network 作为双状态环境

RC合约的另一个关键特性：它们在Reactive network和ReactVM中以两个具有独立状态的实例存在。理解这一概念对于RC合约至关重要。

3.1.1 ReactVM 和 Reactive network之间的区别

每个RC合约都有两个实例，一个位于Reactive network，另一个在ReactVM 中运行。是一个为了**高性能**而设计的特殊系统架构。核心思想是：**把一份智能合约拆成两个“分身”，让它们分工合作，就像工厂的流水线一样。**

为什么要把一个合约做成两个实例？

这完全是为了速度和效率。

分工明确，互不干扰：总控系统（Reactive Network）专心处理事件调度和消息传递；工作站（ReactVM）专心执行计算。两者并行工作，不会因为一方忙碌而卡住另一方。

应对海量事件：当短时间内有大量事件（比如交易、数据更新）涌来时，这个架构就像一条高效的流水线。总控系统快速分拣事件，同时唤醒多个对应的工作站同时开工，从而极大地提升了整体处理能力。

简单来说：

Reactive Network 是协调者，负责调度和通信；ReactVM 是执行者，负责运行合约代码。把一份合约同时放在这两个地方运行，是一种通过“并行化”来提升系统性能的聪明设计。

Reactive Network 的运行方式与典型的 EVM 区块链类似，但它增加了系统合约，允许用户订阅和取消订阅以太坊、BNB、Polygon 或 Optimism 上的源链事件。每个部署地址都拥有一个专用的 ReactVM。

ReactVM 是一个受限虚拟机，旨在隔离地处理事件。从一个地址部署的合约在同一个 ReactVM 中执行。它们可以相互交互，但不能与响应式网络上的其他合约交互。

ReactVM 中的合约可以通过两种方式与外部世界交互，这两种方式都通过Reative network实现：

-   它们会对订阅的特定事件做出反应，并在这些事件发生时执行。
    
-   ReactVM 根据事件输入执行代码，向Reative network发送请求，以获取回调到目标链，从而执行相应的链上操作。

2.1 事件与回调的工作原理

在以太坊中，事件使智能合约能够通过在满足特定条件时记录特定信息来与外部世界通信。这使得去中心化应用程序（dApp）无需持续轮询区块链即可触发和响应事件。事件由以太坊虚拟机（EVM）索引，使其易于搜索，这对于监控区块链活动（例如转账、合约更新和预言机提供的价格变动）尤其有用。

2.1.1 EVM事件

事件是一种**廉价的、不可篡改的、用于对外广播“发生了什么”的日志**。它不负责“做”，只负责“说”，但“说”出来的话可以被无数“耳朵”听到并做出反应。

当智能合约发出事件时，事件数据会被记录在日志中，永久上链。发事件不会直接改变合约的变量状态。但它提供了一种基于事件参数记录和检索信息的方法。

开发者用even关键字定义事件名称，以及记录数据的模版。触发事件时，智能合约用关键字emit接事件名称和要记录的数据，发出记录次事件。

pragma solidity ^0.8.0;

contract DestinationContract {

//定义合约

//用关键字event定义事件

event CallbackReceived(uint256 indexed id, string message);

//回调处理函数

function handleCallback(uint256 id, string memory message)

external {

//用关键字emit发出此事件

emit CallbackReceived(id, message);

// 这里可以添加额外的逻辑来处理回调。

}

}

当事件被发出后，外部应用程序（前端网页，后端服务，RC）可以监听到此事件。程序监听到后，可以自动触发后序操作。（通过指定事件签名以及（可选的）过滤参数，这些应用程序可以在事件发生时订阅实时更新。）这种机制对于创建响应迅速且交互性强的区块链应用程序至关重要。

2.1.2 RC合约中的事件处理

RC合约必须实现IReactive接口来处理传入的事件。

pragma solidity >=0.8.0; import ‘./IPayer.sol’; //IReactive接口继承了 IPayer的功能（通过 is IPayer）。 //IPayer接口通常定义了支付Gas费或处理资金的相关函数， //是响应式合约能“自主支付”的基础。 interface IReactive is IPayer { //用于封装从区块链上捕获到的一个完整事件日志。 //当响应式合约订阅的某个事件发生时，Reactive Network 会将这些详细信息打包成一个 LogRecord实例，传递给合约处理。 struct LogRecord { uint256 chain\_id; address \_contract; uint256 topic\_0; uint256 topic\_1; uint256 topic\_2; uint256 topic\_3; bytes data; uint256 block\_number; uint256 op\_code; uint256 block\_hash; uint256 tx\_hash; uint256 log\_index; } //发起跨链回调的标准事件 event Callback( uint256 indexed chain\_id, address indexed \_contract, uint64 indexed gas\_limit, bytes payload ); function react(LogRecord calldata log) external; }

日志记录结构：一种结构化数据类型，LogRecord用于包含有关事件日志的详细信息：

-   chain\_id：事件发生的区块链 ID。
    
-   \_contract：发出该事件的合约的地址。
    
-   topic\_0至topic\_3：日志的索引主题。
    
-   data：事件日志中未建立索引的数据。
    
-   block\_number：事件发生的街区编号。
    
-   op\_code：可能表示操作码。
    
-   block\_hash、tx\_hash和log\_index：用于追踪事件来源和上下文的其他标识符。
    

回调事件：用于通知订阅者特定事件发生的事件：

-   chain\_id：事件的区块链 ID。
    
-   \_contract：合约发行地址。
    
-   gas\_limit：回调函数分配的最大 gas 值。
    
-   payload：随回调一起返回的编码数据。
    

react 函数：处理传入事件通知的关键函数。

-   接受一个LogRecord输入，使响应式合约能够动态处理事件日志。
    
-   标记为external，仅允许从合约外部调用。
    

RC network持续监控事件的日志，并将合约中定义的条件进行匹配，符合条件时，网络触发相应的react（）方法，并且传递相关详细信息。

RC合约可以访问所有标准的EVM功能。但是，它们运行在私有的ReactVM环境中，这限制了它们只能与同一部署者部署的合约进行交互。这种隔离机制确保响应式合约在处理来自响应式网络的事件时，能够维持一个受控且安全的环境。

2.1.3回调到目标链

RC合约可以t通过特定的格式发出日志记录，在目标链发起交易。这些记录会被响应式网络接收，然后响应式网络会在相关的链上执行所需的交易。

整个过程可以被概括为一个三步的自动化管道：

\[RC 合约\] – 1.发出 Callback 事件 --> \[响应式网络\] – 2.中继与处理 --> \[目标链\] – 3.执行交易 --> \[目标合约\]

发出回调事件

要请求在目标链上执行操作，用户必须Callback在响应式合约中触发一个事件。触发后，智能合约会发出此事件，并提供响应式网络创建和提交交易所需的关键信息。

event Callback( uint256 indexed chain\_id, address indexed \_contract, uint64 indexed gas\_limit, bytes payload );

-   chain\_id：目标网络的 EIP155 链 ID。
    
-   \_contract：合同目的地地址。
    
-   gas\_limit：目标链上交易的 gas 限制。
    
-   payload：用于指定目标函数调用的编码数据。此数据指导响应式网络如何对目标合约执行预期操作。
    

处理回调

当Callback事件触发时，响应式网络会检测到它并处理该事件，该事件会将交易详情编码为特定格式。然后，响应式网络使用提供的和 payload将交易提交到目标链上指定的合约。chain\_idgas\_limit

目标链上的合约执行该交易，完成RC所期望的操作（例如，在Uniswap上执行兑换、转移资产、更新状态等）。

**构造 payload**:

-   payload是指导“在目标合约上具体调用哪个函数、传入什么参数”的编码数据。
    
-   使用 abi.encodeWithSignature等标准Solidity工具进行编码，格式与普通交易调用数据完全相同。
    
-   关键安全机制：出于安全目的，响应式网络**会自动**将 payload中编码的第一个参数（前160位）替换为**调用此RC的ReactVM的地址**。这意味着，在RC编码时，第一个参数位置应预留（如设为 address(0)），它将在网络层被替换为真实的、经过验证的发送者身份。

1.1 Reactive Network 概述

Reactive Network 是一个与 EVM 兼容的链，它围绕 Reactive Contracts (RC) 构建——RC 是一种事件驱动的智能合约，用于跨链和链上自动化。

响应式合约监控EVM链上的事件日志，并在满足条件时自动执行Solidity逻辑。响应式合约无需等待用户或机器人触发交易，而是持续运行并决定何时发送跨链回调交易——本质上**为智能合约提供了“如果……那么……”的自动化功能**。

这使得构建条件性跨链工作流成为可能，例如：

-   自动止损和止盈订单
    
-   清算保护
    
-   自动投资组合再平衡
    
-   产量优化
    
-   跨链工作流程
    

1.2 响应式合同是什么？

1.2.1 响应式

-   响应式合约（RC）代表着我们与区块链技术交互方式的一次范式转变。与由用户交易运行的传统智能合约不同，响应式合约会主动监控与 EVM 兼容的链上的事件并做出响应。它们根据已实现的逻辑处理这些事件，并在区块链上自主执行后续操作。这项创新引入了一种去中心化的机制，无需人工干预即可自动响应链上事件。
    

1.2.2 为什么需要响应式

在以太坊世界中，智能合约彻底改变了我们执行无需信任协议的方式。传统上，这些合约只有在用户发起交易后才会生效。这存在固有的局限性。首先，智能合约无法在没有外部指令（无论是来自用户还是像交易机器人这样的自动化脚本）的情况下自主发起操作或响应区块链事件。这就需要持有私钥并引入中心化的控制点。

响应式合约（RC）的出现正是为了解决这一限制。RC旨在自主响应以太坊虚拟机（EVM）中的事件，并触发区块链生态系统中的后续操作。这种能力使得复杂的逻辑得以实现，它可以从多个链获取信息，并在无需中央监管的情况下跨平台执行更改或交易

1.2.3 响应式合约的优势

-   去中心化：RC在区块链上独立运行，消除了中心化的控制点，并通过降低操纵或故障的风险来提高安全性。
    
-   自动化：RC 会根据链上事件自动执行智能合约逻辑，实现高效的实时响应
    
-   跨链互操作：RC可以与多个区块链进行交互
    
-   增强效率和功能：RC 提高了智能合约的效率，支持复杂金融工具、动态 NFT 和创新 DeFi 应用等高级功能。
    
-   DeFi 及其他领域的创新： RC 为 DeFi 和其他区块链应用带来了新的可能性，例如自动化交易和动态治理，从而创建了一个更具响应性和互联性的区块链生态系统。
    

1.3 响应式合约总结

RC代表了区块链技术的一项重大飞跃，无需用户干预，可以自主执行逻辑。与传统的EVM不同，RC可以自动响应来自多个区块链的事件，从而实现灵活的跨链交互。

反应式合约（RCs）与传统EVM智能合约的核心区别在于**执行范式从“被动响应”转变为“主动监听”**。具体对比如下：

| 对比维度 | 传统EVM智能合约 | 反应式合约 | | 触发方式​ | 被动：必须由外部账户（EOA）或合约账户主动发送一笔交易来调用。 | 主动：持续监听链上事件（如代币转账、状态变更），条件满足时自动触发执行。 | | 执行模式​ | 请求-响应：像一台自动售货机，有人投币（发送交易）才工作。 | 事件驱动：像一个7x24小时在线的自动化机器人，监控环境并自动行动。 | | 持续性​ | 无状态（在交易执行期间）：仅在处理交易时运行，执行结束后即“休眠”。 | 有状态（在事件周期内）：作为后台进程持续运行，等待特定事件。 | | Gas支付方​ | 由发起交易的用户账户支付。 | 通常由RCs的部署者或维护者预付Gas，或通过类似“Gas Station Network”的机制处理。 | | 主要设计目标​ | 实现确定性的业务逻辑和状态变更。 | 实现跨合约、跨区块甚至跨链的复杂自动化工作流。 | | 典型应用​ | 代币发行、去中心化交易所、简单众筹。 | 自动清算、限价订单、跨链桥接、DAO自动化治理、游戏事件响应。 |

**简单来说：**

-   传统合约是**工具**，需要你手动使用。
    
-   反应式合约是**助理**，在你设定的规则下替你自动完成工作。
    

这种区别使得RCs能构建出更复杂、更智能、更自动化的去中心化应用，但同时也带来了更高的设计复杂度和Gas成本管理挑战。
