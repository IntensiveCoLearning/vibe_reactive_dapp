### 1.Origin 合约：抛出带 `indexed` 的稳定事件

目标要求：只负责 `emit Event(...)`，且参数至少包含一个 `indexed`。

在 Insider-hunter 的场景里，我们要监听的是某个特定 DEX 上的交易。`indexed` 的作用极大：它允许我们在底层直接作为 `Topic1` 或 `Topic2` 进行硬件级过滤，而不是把所有日志拉进虚拟机里再用代码判断（那样太费钱了）。

Solidity

```
// Origin Chain (例如 Base Sepolia)
contract MockDEX {
    // 目标事件：注意 insider 加了 indexed，它将成为 Topic 1
    event TargetBought(address indexed insider, address token, uint256 amount);

    function swapTargetToken(address token, uint256 amount) external {
        // ... 真实的 swap 逻辑 ...
        
        // 抛出事件供 Reactive 监听
        emit TargetBought(msg.sender, token, amount);
    }
}
```

### 2\. Reactive 合约：订阅、解析与防抖过滤

目标要求：确认系统合约地址为 `0x...fffFfF`，建立订阅，并在 `react` 中解析 topic/data，做条件判断（去重），最后产出可观测事件。

这里是整个中枢的心脏。 睿应层的订阅机制正是通过比对 ChainID、合约地址以及 Topics（0到3）来实现精准捕获的。

Solidity

```
// Reactive Chain (Lasna Testnet)
import './AbstractReactive.sol';
import './ISystemContract.sol';

contract InsiderHunterReactive is AbstractReactive {
    // 官方指定的系统合约固定地址
    address constant SYSTEM_CONTRACT = 0x0000000000000000000000000000000000fffFfF;
    
    // Topic0: TargetBought(address,address,uint256) 的 Keccak256 哈希
    uint256 constant BOUGHT_TOPIC_0 = 0x...; 
    
    // 记录已经处理过的交易，用于防抖去重
    mapping(bytes32 => bool) public processedLogs;

    // 我们为了前端和后端观测，专门定义一个内部事件
    event HuntTriggered(uint256 originChainId, address insider, uint256 amount);

    constructor(
        uint256 _originChainId,
        address _originDEX,
        uint256 _targetInsiderAddress // 我们要精准狙击的那个内幕钱包地址 (Topic 1)
    ) {
        if (!vm) {
            // 建立订阅：精准匹配 ChainID + 合约地址 + 事件签名(Topic0) + 特定内幕地址(Topic1)
            ISystemContract(payable(SYSTEM_CONTRACT)).subscribe(
                _originChainId,
                _originDEX,
                BOUGHT_TOPIC_0,
                _targetInsiderAddress, // Topic 1：只监听这个人的操作！其他人的不管
                REACTIVE_IGNORE,
                REACTIVE_IGNORE
            );
        }
    }

    function react(LogRecord calldata log) external vmOnly {
        // 1. 去重逻辑：防止同一个事件被重入或重复触发
        bytes32 logId = keccak256(abi.encode(log.chain_id, log._contract, log.topic_0, log.topic_1, log.data));
        require(!processedLogs[logId], "Event already processed");
        processedLogs[logId] = true;

        // 2. 解析日志
        // Topic1 是 insider 地址（由于是 indexed，它在 topics 数组里）
        address insider = address(uint160(log.topic_1));
        
        // Data 里存放的是非 indexed 的参数 (token, amount)
        (address token, uint256 amount) = abi.decode(log.data, (address, uint256));

        // 3. 策略判断：比如金额大于某个阈值才跟单
        if (amount > 1000 ether) {
            
            // 4. 发射可观测事件！这非常关键，你的 Node.js 后端就是盯着这个事件的
            emit HuntTriggered(log.chain_id, insider, amount);

            // 5. 构造回调，发射去目标链的 Callback
            bytes memory payload = abi.encodeWithSignature(
                "executeHunt(address,address,uint256)", 
                address(0), // 占位符，详见目标合约解释
                token, 
                amount
            );
            emit Callback(DESTINATION_CHAIN_ID, DESTINATION_CONTRACT, GAS_LIMIT, payload);
        }
    }
}
```

### 3\. Destination 合约：权限校验的“守门员”

目标要求：回调入口函数，强制**第一个参数为** `address`。

为什么必须是 address？在 Ivan 的讲解中提到，这是 Reactive Network 为了安全（Authorization）做的硬性规定。当 Relayer 替你调用目标链时，它会自动把**部署该 Reactive 合约的 RVM ID (也就是你的部署钱包地址)** 塞进第一个参数里。这样目标合约就能确认：这个指令真的是“我自己的”监控程序发出的，而不是别人乱调用的。

Solidity

```
// Destination Chain (例如 Ethereum Sepolia)
contract HunterExecution {
    address public myRvmDeployerAddress;

    constructor(address _myRvmDeployerAddress) {
        myRvmDeployerAddress = _myRvmDeployerAddress;
    }

    // 第一个参数必须是 address rvmId
    function executeHunt(address rvmId, address token, uint256 amount) external {
        // 权限校验：确保只有我自己的 ReactVM 实例能触发这个跟单动作
        require(rvmId == myRvmDeployerAddress, "Unauthorized RVM");
        
        // ... 在目标链上执行真正的买入跟单逻辑 ...
    }
}
```

ETHPanda 往期 Workshop 视频中，Ivan（DevRel 负责人）花了很大篇幅解释了这套系统的双状态隔离机制和延迟处理。结合官方要求和架构师思维，我们将这三大模块（智能合约、后端、前端）拆解为一个协同工作的系统：

### 1\. 智能合约层（The Engine：精密过滤与隔离）

在 Hackathon 中，评委看重的是边缘情况的处理和逻辑的严密性。

-   **双状态模型深入 ：** 就像视频里强调的，部署后会有两份实例。`Reactive Network 实例` 负责在 `constructor` 中调用固定的系统合约（`0x...fffFfF`）交钱并建立 `subscribe`；而 `ReactVM 实例` 才是真正执行 `react(LogRecord)` 的地方。两者的状态是隔离的！
    
-   **精准狙击 (Indexed 匹配)：** 在 Origin 合约中，你抛出的 Event 必须带有 `indexed` 参数。比如对于 Insider-hunter，`Topic0` 是交易事件签名，`Topic1` 就可以是你要追踪的那个“内幕钱包地址”。这样底层的过滤维度就能做到精确匹配，不浪费计算资源。
    
-   **回调与防抖：** 在 `react` 函数中不仅要解析日志，还要加入**去重逻辑（防重入/重复触发）和白名单验证**，最后一定要 `emit` 一个可观测的内部事件，这是给你的后端留下的“钩子”。
    

### 2\. 后端中间件（The Watchtower：全知观测者）

你可能会问：既然都在链上自动跑了，为什么还需要后端？ 答案是：**为了极致的用户体验（UX）。** 跨链操作通常有 10-15 秒的延迟，如果前端没有反馈，用户会以为卡死了。

-   **双链监听：** 你的 Node.js 或 Rust 后端需要同时开启两个 RPC 监听。一个盯着 Origin 链（比如 Base Sepolia），一个盯着 Lasna 链（读取你合约 `emit` 的内部排错事件）。
    
-   **WebSocket / SSE 推送：** 后端将捕捉到的散落事件拼装成一个结构化的 JSON，通过长链接实时推给前端。
    
-   **排错指引：** 后端还要负责抓取 ReactVM 的具体地址和 Transaction Hash，以便在 UI 上直接生成前往 Reactscan 的一键跳转链接。
    

### 3\. 前端界面（The Command Center：让魔法可视化）

Hackathon 的胜负往往取决于 Demo 的可视化程度。Reactive 的跨链过程在底层是“隐形”的，前端的任务就是把它显现出来。

-   **环境控制：** 能够连接钱包，并在 Origin 链（触发事件）和 Lasna 链（查看/充值 Reactive 合约）之间丝滑切换。
    
-   **核心组件【三段式时间线】：** 这是一个极佳的 UI 灵感。你可以在界面上做一个动态的进度条：
    
    1.  🟢 _Origin 事件已触发 (Tx Hash)_ ->
        
    2.  🟡 _Reactive 捕获计算中 (等待 Lasna 确认)_ ->
        
    3.  🔵 _Destination 回调成功 (目标链抛出结果)_。

**双状态模型（Dual-State Model）**。需要把睿应合约想象成具有“两个人格”：

-   **人格一：网络态 (Network State) —— 负责收发指令与计费**
    
    -   **何时触发：** 当部署合约（`constructor`）或者外部账户直接调用合约时。
        
    -   **能做什么：** 它处于“正常”的 EVM 环境中。在这里调用 `!vm` 包裹的代码，比如告诉底层的 Ingress 网络“我要订阅某个 DEX 的大额转账事件”，或者往合约里充值 Gas 费。
        
    -   **不能做什么：** 它**不能**处理系统自动推送过来的日志。
        
-   **人格二：执行态 (VM State) —— 负责核心的量化与判断逻辑**
    
    -   **何时触发：** 只有当 Ingress 真的捕获到了订阅的事件日志，并将其推送给 ReactVM 时，系统才会瞬间“唤醒”这个状态。
        
    -   **能做什么：** 它通过 `vmOnly` 修饰的 `react()` 函数来运行。在这里，可以写极其复杂的计算逻辑（例如：分析日志中的买入量，结合当前流动性池深度，计算出滑点和的 Insider-hunter 策略是否应该跟单）。
        
    -   **不能做什么：** 它处于一个沙盒中，**它无法直接修改网络态的持久化存储变量**（这是为了极高并发下的数据一致性）。如果它计算完毕决定要执行跟单，它唯一的输出方式就是 `emit Callback`。
        

### 跨链与自动化机制：

ReactVM 经过精密计算，在 Kopli 测试网（或未来的主网）的 `react()` 函数里发射了一个 `Callback` 事件：“在以太坊主网上买入 10 个 ETH！”

但问题来了：Kopli 链上的一个事件，是怎么变成以太坊主网上的一笔真实交易的？

这就引出了跨链自动化的核心机制：**Relayer（中继器网络）**。

-   **第一步：信号拦截 (Emission)** 睿应合约发射了包含具体指令（Payload）和目标地址的 `Callback` 事件。这个事件被记录在睿应层的区块日志中。
    
-   **第二步：中继器抢单 (Relayer Network)** 在睿应链下，有一群被称为 Relayer（中继器）的去中心化节点。它们的作用类似于以前的“清算机器人”。它们死死盯着睿应链上的 `Callback` 事件。
    
-   **第三步：构造跨链交易 (Transaction Formatting)** 一旦某个 Relayer 看到了 `Callback` 信号，它会立刻提取出要执行的 Payload（比如调用目标链上回调合约的 `sellTokens` 函数）。
    
-   **第四步：代付 Gas 与执行 (Execution & Automation)** **这是自动化的关键！** Relayer 会使用**它自己**在以太坊主网上的钱包账户和私钥，打包这笔包含 Payload 的交易，并支付以太坊主网高昂的 Gas 费，然后提交上链。
    
-   **第五步：结算 (Settlement)** 天下没有免费的午餐。Relayer 在目标链垫付了真金白银，它会拿着以太坊上执行成功的凭证，回到睿应层，从预先存放在睿应合约里的资金池中扣除相应的费用（并赚取微小的服务费）。

### Subscribe（订阅与过滤）：部署“雷达”

在一个止损单（Stop Order）中，首先需要知道 DEX 里代币的价格变化。Uniswap V2 每次发生交易时，流动性池（Pair）都会抛出一个名为 `Sync(uint112 reserve0, uint112 reserve1)` 的事件。

在合约的初始化阶段，我们需要把“雷达”架设好：

```
// Uniswap V2 Stop Order 示例（精简版）
contract UniswapV2StopOrder is AbstractReactive {
    // Sync 事件的哈希值 (Topic 0)，用来精确过滤日志
    uint256 constant SYNC_TOPIC_0 = 0x1c411e9a96e071241c2f21f7726b17ae89e3cab4c78be50e062b03a9fffbbad1;
    
    constructor(address _service, address _pair) {
        // 【双态模型初探】：!vm 确保这段代码只在网络态执行，向 Ingress 注册监听
        if (!vm) {
            ISystemContract(payable(_service)).subscribe(
                ORIGIN_CHAIN_ID,
                _pair,            // 监听特定的流动性池合约
                SYNC_TOPIC_0,     // 仅过滤 Sync 事件
                REACTIVE_IGNORE, 
                REACTIVE_IGNORE,
                REACTIVE_IGNORE
            );
        }
    }
}
```

这里的 `SYNC_TOPIC_0` 就是过滤噪音的滤网。如果以太坊上每秒有一万个事件，Ingress 节点只会把匹配这个哈希的事件“捞”出来。如果以后要监听某个特定的内幕钱包地址，就会把那个地址作为 `Topic 1` 或 `Topic 2` 传进去进行精准狙击。

这几个学习目标其实可以串联成一个完整的**高级交易流生命周期**：

### 1\. 订阅与过滤 (Subscribe & Trigger)

在第一阶段，我们只是简单地监听了一个事件。但在实际的自动化交易中，链上噪音巨大。

-   如何使用 `topic0, topic1, topic2` 进行精确过滤？
    
-   例如：如果你在构建一个寻找内幕交易的猎手工具，你不仅需要监听“交易发生（Topic0）”，还要精准锁定“特定交易对（Topic1）”或“特定大户地址（Topic2）”。
    

### 2\. 状态隔离与计算 (ReactVM 双状态模型)

我们在第一阶段简单提到了 `!vm` 和 `vmOnly`。这个“双态模型（Dual-State）”是睿应层最反直觉但也最天才的设计。

-   **网络态 (!vm)**：负责向 Ingress 支付费用、注册和管理订阅规则。
    
-   **执行态 (vmOnly)**：当 Trigger 发生时，真正的计算逻辑（计算滑点、判断套利空间）在这里执行。 理解这两者如何隔离且共享数据，是写出安全睿应合约的关键。
    

### 3\. 跨链执行与经济驱动 (Callback & Economy)

-   **Callback**：计算完成后，如何安全地将包含复杂 Payload 的指令发送到目标链。
    
-   **经济模型 (Economy)**：天下没有免费的午餐。中继器（Relayer）在目标链（比如以太坊主网）帮你发交易是需要消耗真金白银 Gas 的。你需要在睿应合约中提前存入资金来维持这个自动化系统的运转。
    

### 4\. 最佳实践参照物 (Uniswap V2 Stop Order)

官方提供的止损单（Stop Order）脚手架是一个极佳的工业级示例。它完整展示了如何监听流动性池、计算价格滑点，并在满足条件时触发目标链的卖出回调。这套代码稍加改造，就能变成非常强大的量化交易底层逻辑。

根据**分解问题**和**从简单到复杂**的原则，我们要构建的跨链系统可以清晰地拆分为三个独立的“零件”：

⚙️ **事件源合约 (Origin Contract)：** 部署在源链（例如 Sepolia 测试网）。它负责被用户调用，并抛出特定的事件日志（Log）。

⚙️ **回调合约 (Destination Contract)：** 部署在目标链。它负责接收来自睿应层 Relayer 的指令并执行最终动作。

⚙️ **睿应合约 (Reactive Contract)：** 部署在 ReactVM 中。它是连接前两者的桥梁，负责 `subscribe`（监听）源合约，并 `react`（回调）目标合约。

### 第一步：准备测试网环境与水龙头（基础设施）

在跨链回调的架构中，通常的业务合约和睿应合约不在同一条链上：

1.  **业务链（比如以太坊 Sepolia 测试网）：** 需要在这里部署 `Origin.sol`（事件源）和 `Destination.sol`（回调目标）。需要准备一点 Sepolia ETH 测试币。
    
2.  **睿应链（Reactive Kopli / Lasna Testnet）：** 需要在这里部署 `BasicReactive.sol`（睿应合约）。需要去官方 Discord 或水龙头领取 Reactive 网络的测试代币（REACT）。

-   **Origin 合约：** 非常简单，里面通常只有一个触发函数，调用后会抛出一个类似 `Ping` 或 `EventEmitted` 的日志。将其部署到 Sepolia，**记录下合约地址 A**。

-   拿着刚才获取的 **地址 A** 和 **地址 B**，作为构造函数参数。
    
-   将 `BasicReactive.sol` 部署到 Reactive 测试网。
    
-   部署成功后，跨链的监听与回调链路就正式打通了！

今天为了深度理解睿应层的底层架构和代码需求，认真剖析整个架构逻辑。明天打算解决一下遗留的问题，完成小挑战第一关。

这里把“三驾马车（Ingress - ReactVM - Egress）”与代码层面的“两行核心指令（Subscribe - React）”进行一次直接映射：

### 1\. `subscribe()`：发送给 Ingress 的“配置指令”

-   **在代码层面**：编写了一行 `subscribe(以太坊, Uniswap合约, Swap事件)`。
    
-   **在底层架构中**：这行代码实际上是在向 Ingress（去中心化监听网络）下达任务。Ingress 收到指令后，其节点便会调整“天线”，开始精准盯盘以太坊上的目标 Uniswap 合约。
    

### 2\. `react()`：ReactVM 内部的“触发器”

-   **在底层架构中**：整个智能合约（包含所有业务逻辑）均部署在 ReactVM（计算核心）内部。当 Ingress 确切监听到链上的 Swap 事件时，便会捕获该事件，并将其推入 ReactVM。
    
-   **在代码层面**：ReactVM 接收到事件后，如何定位需要执行的合约片段？这正是 `react()` 发挥作用的时刻。ReactVM 会将事件数据精准注入 `react()` 函数中，进而唤醒相应的业务逻辑（例如计算价差）。
    

### 3\. `react()` 的输出：交由 Egress 负责“跑腿”

-   **在代码层面**：`react()` 函数经过逻辑运算后，若发现存在套利空间，便会在代码末尾触发指令：“前往目标链发起一笔买入交易”。
    
-   **在底层架构中**：该指令一旦由 `react()` 函数发出，即刻会被 Egress（中继器出口网络）接管。Egress 负责打包这笔交易，垫付 Gas 费用，并最终将其发送至目标链上落地执行。
    

* * *

> **💡 通俗类比（以智能家居场景为例）：**
> 
> -   **Ingress - ReactVM - Egress** 宛如一套完整的智能硬件系统（摄像头 -> 中控主机 -> 智能门锁）。
>     
> -   **Subscribe / React** 则类似于控制端 App 内预设的自动化规则（开启车库监控 -> 若检测到访客，则自动锁死大门）。
>     
> 
> 一旦设定好规则（Subscribe/React），硬件系统（Ingress/Egress）便会依循这些规则在物理世界中运转。若脱离底层架构，再精妙的代码也只是纸上谈兵；反之，若没有代码指令的引导，底层架构也将失去监听与执行的方向。

# 2026-03-11
<!-- DAILY_CHECKIN_2026-03-11_START -->









为了实现跨链监听，Reactive Network 采用了一种叫做 **控制反转（Inversion of Control）** 的设计，把它的架构拆解为三个核心步骤：

## 1\. Ingress（入口）：去中心化的“千里眼”

在传统的开发中，如果要监听链上事件，需要自己在 VPS 上部署节点或通过 RPC 接口不断轮询（Polling）拉取区块日志。 在睿应层中，这个脏活累活由 **Reactive Nodes（睿应节点网络）** 接管了。

-   这些节点会持续同步并索引多个源链（Origin Chains，比如以太坊、BSC 等）的区块数据。
    
-   当源链上的某个合约（比如 DEX）发生交易并抛出 **Log（日志/事件）** 时，睿应节点会立刻捕获到这个 Log。
    

## 2\. ReactVM（计算核）：事件驱动的中枢

这是睿应层的“大脑”。你编写的 **睿应式智能合约（Reactive Smart Contract）** 就部署在这里，而不是部署在以太坊或 BSC 上。

-   你的睿应合约里会写明：“我要**订阅（Subscribe）** BSC 链上合约 A 的特定主题（Topic）日志”。
    
-   当 Ingress 捕获到匹配的日志后，它会把这个日志作为参数，**主动喂给** ReactVM 中的睿应合约。
    
-   ReactVM 接收到日志后，瞬间唤醒你的合约代码，执行写好的逻辑（比如，计算刚刚发生的交易滑点是否满足套利模型）。
    

## 3\. Egress（出口）：跨链执行的“机械臂”

如果 ReactVM 计算后发现条件满足（比如，确实存在套利空间），它需要去目标链（Destination Chain）采取行动。

-   睿应网络会通过 **Relayers（中继器网络）** 构造一笔标准的以太坊/EVM 交易。
    
-   网络使用共识机制或阈值签名，授权这笔交易，并向目标链广播。
    
-   最终，目标链上的合约被触发，完成了想要的自动化操作
    

不再需要维护复杂的链下服务器架构、Docker 容器或担心 RPC 节点挂掉。

只需要写一段 Solidity 代码，告诉 ReactVM “**监听什么链的什么事件 -> 做出什么判断 -> 去什么链执行什么操作**”，网络底层的节点就会自动完成跨链的数据搬运和触发。

# 2026-03-10

## 1\. 传统模式：依赖中心化的“链下机器人”

目前 EVM 无法自己行动。为了捕捉滑点和价差，开发者必须在自己的 VPS 或云服务器上部署一个自动化脚本（比如用 Rust 或 Python 写的机器人）。

-   **工作流：** 机器人（链下）不断轮询节点获取最新区块数据 -> 发现价差 -> 机器人计算利润 -> 机器人使用自己的私钥（EOA 账户）向目标链发起一笔包含 Gas 的交易。
    
-   **痛点：** 存在单点故障（你的服务器可能会宕机，RPC 节点可能会限流），并且这是一种“Web2 监听 Web3”的中心化方式。
    

## 2\. 睿应层模式：链上原生的“事件驱动”

有了睿应层，可以直接部署一个 **睿应式智能合约（Reactive Smart Contract）**。这个合约生存在 ReactVM 中。

-   **工作流：** 在合约里写好逻辑：“订阅 BSC 链上 PancakeSwap 的 `Swap` 事件。如果计算出价格与以太坊 Uniswap 的差值大于 X，则触发以太坊上的买入操作。”
    
-   **结果：** 当 BSC 上发生大额交易导致价格剧烈波动时，ReactVM 的节点网络会捕获这个事件日志（Log），并**自动、去中心化地**在目标链上执行你的套利代码。完美绕过了自己维护链下服务器和机器人的繁琐工作。  
      
    **睿应层的真正杀手锏是“去中心化的自动化（Decentralized Automation）”与“抗审查”。** 它把原本脆弱的链下机器人逻辑，变成了永不宕机的链上协议。它非常适合那些需要极高可靠性、跨链状态同步，以及无需信任的自动化工作流（比如自动化复投、清算保护、或是复杂的跨链资金管理）。

# 2026-03-09
<!-- DAILY_CHECKIN_2026-03-09_START -->











1.  今日学习了睿应层（Reactive Network）与传统 EVM 的根本区别，这种转变将智能合约从“静态工具”升级为了具备实时响应能力的“**主动参与者**”。
    
2.  可归纳为以下核心点：
    

-   **执行驱动力（控制反转）：** 传统 EVM 是“**被动执行**”的，智能合约必须由外部账户（EOA）发起交易并支付 Gas 才能运行。睿应层引入了“**控制反转（Inversion-of-Control）**”，其反应式合约（RSC）可主动监听并响应链上事件，实现无需外部干预的**自主执行**。
    
-   **交互范式（事件驱动）：** EVM 采用“请求-响应”线性模式。而睿应层采用**事件驱动架构**，将“感知”与“执行”分离。它像一个持续运行的“If-This-Then-That”自动化引擎，能根据预设条件实时捕捉多链数据流并触发动作。
    
-   **跨链协同（全链编排）：** 传统 EVM 合约是孤立的，跨链需依赖第三方桥或预言机。睿应层作为**全链异步层**，原生支持跨链日志监听，能在一个逻辑中指挥多条 EVM 链的状态变更，消除了对中心化机器人或离线脚本的依赖。
    
-   **性能架构（并行执行）：** 区别于以太坊顺序处理交易的串行逻辑，睿应层通过 **ReactVM** 实现**并行化 EVM 执行**，支持多线程独立工作，极大提升了处理大规模事件流的效率与吞吐量。
