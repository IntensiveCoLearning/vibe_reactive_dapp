## **一、核心概述**

Subscriptions 用于定义**Reactive Contracts (RCs，反应式合约)** 监听的事件范围，RCs 通过系统合约订阅事件，需指定**源链ID、合约地址、事件主题（Topics）** 三个核心维度；当检测到匹配事件时，合约的`react()`函数会被触发。

订阅的配置方式分为两种：

1.  部署阶段：在合约构造函数中配置；
    
2.  运行阶段：通过回调（callbacks）动态创建/删除。
    

## **二、订阅基础**

### **1\. 核心操作**

通过调用**系统合约**的`subscribe()`方法创建订阅，该操作通常在合约构造函数中执行。

### **2\. 关键注意点**

合约会同时部署在**Reactive Network (RNK)** 和**ReactVM**中，而ReactVM内不存在系统合约，因此构造函数中需通过`if (!vm)`判断，避免在ReactVM中调用`subscribe()`。

### **3\. 必要实现接口**

需实现`IReactive`、`AbstractReactive`、`ISystemContract`三个接口，才能完成订阅功能开发。

### **4\. 事件过滤维度**

订阅通过以下维度过滤事件，精准匹配需要监听的内容：

-   链ID（Chain ID）
    
-   合约地址（Contract address）
    
-   事件主题0-3（Topics 0–3）
    

## **三、通配符与匹配规则**

Reactive Network 提供通配符实现**模糊匹配**，但要求**至少有一个参数为具体值**，不允许全通配符。

### **1\. 预定义通配符：REACTIVE\_IGNORE**

-   作用：匹配**任意主题值**；
    
-   固定值：`0xa65f96fc951c35ead38878e0f0b7a3c744a6f5ccc1476b313353ce31712313ad`。
    

### **2\. 零值通配符**

-   `uint256(0)`：匹配**任意链ID**；
    
-   `address(0)`：匹配**任意合约地址**。

## **六、订阅限制（Subscription Limitations）**

### **1\. 仅支持精确匹配**

不支持范围、比较、位运算等模糊过滤，仅能通过**等值匹配**筛选事件，不支持：

-   大小比较（`<`/`>`）；
    
-   数值范围；
    
-   位运算过滤。
    

### **2\. 单订阅仅支持单一筛选规则**

不支持在一个订阅中设置**或（OR）条件**、多套筛选规则，解决方案为：多次调用`subscribe()`创建多个订阅（可能导致订阅数量过多）。

### **3\. 不允许全局订阅**

禁止创建覆盖全链、全合约、全事件的全局订阅，即不允许：

-   匹配所有链；
    
-   匹配所有合约；
    
-   匹配某条链上的所有事件。
    

### **4\. 重复订阅的处理**

允许创建重复订阅，但实际效果**等同于单个订阅**；且每次`subscribe()`交易仍会消耗Gas，需避免重复创建。

## **七、动态订阅（Dynamic Subscriptions）**

支持根据**入站事件**动态创建/删除订阅，因ReactVM内无法直接调用系统合约，需通过**回调交易**完成订阅管理。

### **1\. 核心执行流程**

1.  ReactVM接收事件；
    
2.  合约判断是否需要创建/删除订阅；
    
3.  合约触发`Callback`事件；
    
4.  Reactive Network (RNK) 执行实际的订阅变更操作。
    

**参数说明**：

-   `rvm_id`：ReactVM标识符，由系统自动注入；
    
-   `subscriber`：需要订阅/取消订阅的地址。
    

## **八、react() 函数核心逻辑**

`react()`是ReactVM合约实例的专属方法，用于**处理入站事件**，并在需要变更订阅时触发回调，核心功能为事件分发与回调触发。

### **1\. 事件处理逻辑**

1.  匹配**订阅主题**：触发`Callback`事件，由RNK执行订阅创建；
    
2.  匹配**取消订阅主题**：触发`Callback`事件，由RNK执行订阅删除；
    
3.  匹配**业务事件**：解析事件数据，触发对应业务回调（如Approval事件处理）。
    

### **2\. 关键特性**

`Callback`事件会在**事件处理完成后**，由Reactive Network统一执行，保证操作的原子性和有效性。

## **九、核心关键词汇总**

| 关键词 | 核心含义 |
| --- | --- |
| Reactive Contract (RC) | 反应式合约，核心订阅主体 |
| ReactVM | 合约运行环境，无系统合约 |
| Reactive Network (RNK) | 主网络，可访问系统合约 |
| REACTIVE_IGNORE | 主题通配符，匹配任意主题值 |
| subscribe() | 系统合约的订阅创建方法 |
| react() | 处理入站事件的核心方法 |
| Callback | 触发动态订阅变更的事件 |
| unsubscribeContract() | 系统合约的取消订阅方法 |

一、极端市场波动下的实时监控机制

核心监控逻辑

监控范围：Reactive 会监控链上所有交易与事件，不仅限于波动相关事件，覆盖全部链上进程。

潜在问题：向目标链发送回调交易时，若交易 / 汇率变化过快，回调可能延迟送达。这是 DeFi 领域的共性问题，并非 Reactive 独有。

优化方向：Reactive 会监控所有区块事件，并提供快速响应能力，尽可能降低延迟影响。

二、止盈止损功能的合约部署优化

部署数量要求

主网环境：至少需部署 2 个合约：

反应式合约：负责响应链上事件。

回调合约：负责与目标链交易池交互。

该数量为功能实现的必要条件，无法进一步减少。

测试网环境：建议自行部署交易对，便于控制资金池流动性、调整汇率，更利于测试。

复用优化方案

预部署可复用的回调合约与监控合约，减少重复部署成本。

B2C 场景（如 Reactify 应用）可代用户部署合约；若能设计合理支付管理方案，实现多用户复用同一合约，将是优质解决方案。

三、「订阅 - 触发 - 回调」模型的平衡与稳定性

1\. 即时响应 vs 执行确定性

即时响应：源链事件一旦确认（甚至零确认），反应式合约的 react 函数立即执行。

执行确定性：需开发者在智能合约层面自行实现，例如采用回调乒乓机制：

反应式合约向源链发送回调，反复确认多次区块后再执行后续操作，规避区块重组风险。

灵活平衡：Reactive 提供基础工具，开发者可通过「等待指定区块数」「用户额外确认」等机制，按需平衡响应速度与确定性。

2\. 高频场景适配性

不适用场景：高频预测市场（如每秒多次更新价格的衍生品市场）。

原因：Reactive Network 出块时间为 7 秒，叠加目标链延迟，整体响应约 10 秒，无法满足高频交易需求。

适用场景：非高频预测市场（如体育赛事结果、选举结果等低频事件投注）。

高频替代方案：若需高频场景，需牺牲去中心化特性，选择中心化方案。

四、Uniswap 止盈止损 Demo 常见问题解决

「余额不足」报错处理

问题原因：执行添加流动性命令前，未先向钱包授权资金池代币，导致钱包无对应代币可操作。

解决方案：

先执行「向资金池代币授权」操作。

再执行「添加流动性」命令。

文档步骤顺序需调整，建议将授权步骤放在添加流动性之前。

Reactive Library 是用于在 Reactive Network 上开发响应式智能合约的基础库，基于 Foundry 构建，提供订阅、回调、支付与系统合约交互能力，安装命令为 `forge install Reactive-Network/reactive-lib`。

库核心是四类抽象合约：AbstractReactive 作为基础合约，提供执行环境区分与系统合约接入；AbstractPayer 处理支付、债务结算与授权发送；AbstractCallback 实现回调权限控制；AbstractPausableReactive 支持订阅暂停与恢复，方便管理链上事件监听。

配套接口定义事件数据结构、订阅与支付标准，其中 IReactive 的 react () 是处理链上事件的入口。系统层由系统合约、回调代理、订阅服务协同，负责费用、权限、事件分发。

库还内置 CRON 定时机制，按区块间隔触发定时事件，支持从每块到万块级周期，无需轮询即可实现自动化任务。整体设计模块化、权限严格，适合开发跨链响应式应用。

### 1\. EVM 整体架构

-   EVM 是**基于栈的 256 位虚拟机**，沙盒、确定性、图灵完备
    
-   执行单元：**PC 程序计数器 + Stack 栈 + Memory 内存 + Storage 存储 + Gas**
    
-   合约代码是只读 “虚拟 ROM”，执行时不修改代码本身
    

### 2\. 三大存储区域（必记）

-   **Stack（栈）**：执行核心，1024 深度，256 位 / 项，所有运算基于栈
    
-   **Memory（内存）**：线性临时内存，字节寻址，交易结束即清空
    
-   **Storage（存储）**：账户级持久化 KV 存储，上链、耗 Gas 高
    

### 3\. 执行流程

1.  交易 / 调用进入 EVM
    
2.  加载字节码 → PC 逐条取指令
    
3.  栈操作 → 内存读写 → 存储读写
    
4.  Gas 扣费 → 状态更新 → 输出结果 / 日志
    

### 4\. 关键特性

-   **确定性**：同输入必同输出，保证全网共识
    
-   **Gas 机制**：计量计算成本，防止滥用
    
-   **无浮点、无原生随机**：保证执行一致性
    
-   指令集（Opcodes）约 140+，覆盖运算、内存、存储、链上信息
    

### 和 Reactive Contract 的关联：

-   EVM 是**执行底座**，Reactive Contract 是**事件驱动的上层范式**
    
-   Reactive VM 兼容 EVM，栈 / 内存 / 存储模型完全一致
    
-   反应式合约的**回调执行、状态更新、跨链消息**，最终都跑在 EVM 规范上

# **学习笔记：Reactive Network 生态案例总结**

**来源**：reactive.network/ecosystem#cases

**主题**：反应式智能合约（RSC）真实应用场景概括

* * *

## **一、核心思想**

Reactive Network 做的是：

**监听链上事件 → 自动触发合约逻辑 → 跨链/多链协同执行**

不用人点、不用中心化服务器、不用托管私钥。

* * *

## **二、生态案例概括**

### **1\. 自动化做市与流动性管理**

-   监听池子价格、深度、费率变化
    
-   自动调整流动性仓位、复利收益、再平衡
    
-   提升资金效率，减少人工操作
    

### **2\. 跨链套利与多链策略**

-   同时监控多条公链的价格差异
    
-   自动执行套利、对冲、搬砖
    
-   去中心化、无单点故障、更安全
    

### **3\. 智能清算与风控自动化**

-   实时监听借贷协议抵押率、健康系数
    
-   达到阈值自动触发清算/追加抵押
    
-   比机器人更快、更去中心化、更公平
    

### **4\. 自动复利收益（Yield Compounding）**

-   监听收益发放、claim 事件
    
-   自动复投、复利滚存、优化收益
    
-   适合流动性挖矿、LSD 等场景
    

### **5\. NFT 自动化工具**

-   监听 NFT 价格、地板价、成交、拍卖
    
-   自动出价、扫货、挂单、归集
    
-   链上自动化，无中心化后台
    

### **6\. DAO 治理自动化**

-   监听投票事件、快照结果
    
-   自动执行提案后的链上操作
    
-   提高 DAO 执行效率，减少人工干预
    

### **7\. 链上事件通知与预警**

-   监听大额转账、合约调用、异常波动
    
-   自动发通知、记录、上链存证
    
-   可对接钱包、DApp、后端服务
    

### **8\. 跨链资产同步与桥接自动化**

-   监听源链存款/锁定事件
    
-   自动在目标链 mint/解锁资产
    
-   提升跨链体验，减少信任成本
    

### **9\. 链上游戏自动化**

-   监听游戏内道具、战绩、任务完成
    
-   自动领取奖励、合成、升级、对战
    
-   规则上链，公平透明
    

### **10\. 预言机与数据联动**

-   监听价格、行情、链下数据上链
    
-   自动触发策略：止盈、止损、再平衡
    
-   适合 DeFi、量化、资产管理
    

* * *

## **三、共同特点**

1.  **事件驱动**：不依赖人工调用
    
2.  **去中心化**：不用中心化服务器/机器人
    
3.  **多链兼容**：一条 Reactive 管多条链
    
4.  **安全**：不托管私钥，逻辑上链透明
    
5.  **自动执行**：7×24 小时无人值守
    

* * *

## **四、学习收获**

-   Reactive 不是公链，是**多链自动化中间层**
    
-   所有案例本质都是：**把“人盯盘+手动操作”变成链上自动程序**
    
-   未来可用于：DeFi 量化、自动理财、跨链工具、游戏、DAO 等

**核心主题**：反应式智能合约的技术原理、风险控制、生态支持与开发实践

## **一、核心概念与定位**

### **1\. 反应式智能合约（Reactive Smart Contracts, RSC）**

-   **定义**：基于 Reactive Network 执行层的事件驱动型智能合约，采用**控制反转（IoC）** 范式，无需用户主动触发，通过监听链上事件自动执行预定义逻辑。
    
-   **与传统合约的核心区别**：传统合约被动等待 EOA 账户调用；RSC 主动监控多链事件（如转账、交易、借贷），满足条件后自动发起跨链回调或状态变更。
    
-   **底层支撑**：Reactive Network 为 EVM 兼容的执行层，通过**并行化 EVM** 实现低成本、高吞吐量的事件驱动计算。
    

### **2\. Reactive Network 核心价值**

-   **多链协同**：作为区块链协调层，统一处理多链事件与计算，实现跨链无缝交互。
    
-   **去中心化自动化**：替代传统“中心化机器人+私钥托管”的自动化方案，消除单点故障。
    
-   **原生可组合性**：支持 DeFi、NFT、AI 代理等多场景模块化集成，实现“即插即用”的工作流搭建。
    

## **二、关键技术问题解析（基于分享会核心问答）**

### **问题1：反应式智能合约的“公开性”与套利策略适配性**

1.  **代码公开性**：无论合约代码是否在以太坊浏览器或 GitHub 开源，**合约字节码均可被获取和反编译**，本质上具备公开性。
    
2.  **策略适配边界**：仅适用于**高频率套利策略**（如跨资金池套利），**不适合抢跑式短线交易**。
    
    -   原因：反应式工作流执行存在约 10 秒延迟（含事件触发、跨链传输、合约执行），无法满足短线交易的毫秒级响应需求。
        

### **问题2：风险控制的核心优势**

在 AI 代理场景下，RSC 的风险控制优势体现在三点：

1.  **降低单点故障**：将自动化逻辑托管于 RSC，AI 代理仅负责触发工作流，无需接触资金或主钱包私钥，仅需掌握触发钱包私钥，风险范围大幅缩小。
    
2.  **安全透明的逻辑执行**：合约逻辑**不可篡改**，AI 代理只能按预设步骤运行，避免越权操作或恶意行为。
    
3.  **跨协议可组合性支持**：在跨链、跨协议场景中，可将去中心化交易所、资金池、流动性协议等组合，通过 RSC 实现协同工作，提升复杂场景的安全性。
    

### **问题3：动态实体（Dynamic Entities）与 FODL 理论适配性**

-   **当前结论**：在反应式智能合约场景下，暂无法明确动态实体与 FODL 理论的具体结合方式，需结合具体业务场景进一步细化问题（如动态实体的链上身份定义、FODL 预言机的数据交互逻辑）。
    

### **问题4：生态支持与公链集成计划**

1.  **Starknet 支持**：**长期规划支持**，但短期内不落地。因 Starknet 具备速度快、数据量大的特点，需针对性开发适配方案，优先满足用户高频需求。
    
2.  **公链集成优先级**：优先支持用户需求集中的公链/二层网络（如 Arbitrum、Optimism），按开发成本与社区需求分阶段落地。
    
3.  **开发者自主集成**：支持开发者自行开发公链适配模块，团队提供技术指导。
    

### **问题5：基于 RSC 开发交易机器人的优势**

1.  **核心优势：安全性**
    
    -   传统交易机器人：需托管私钥，存在黑客攻击、程序漏洞导致的资金损失风险。
        
    -   RSC 交易机器人：基于链上事件触发，合约仅执行预设逻辑，**无越权操作可能**，风险仅源于代码本身的安全性。
        
2.  **核心能力**
    
    -   支持**定时触发**（如按区块高度、时间间隔）与**事件触发**（如价格波动、流动性变化）。
        
    -   可实现跨链交易自动化，无需人工干预或中心化桥梁。
        
3.  **与 Unikn 的区别**：Unikn 侧重链上身份与权限管理，RSC 侧重事件驱动的自动化执行，二者可结合实现“身份认证+自动化操作”的复合场景。
    

### **问题6：跨链转账失败的处理机制**

1.  **失败核心原因**：跨链操作涉及“源链确认+目标链执行”两个阶段，任何阶段的网络拥堵、合约错误、gas 不足均会导致失败。
    
2.  **官方处理方案（核心步骤）**
    
    1.  监控源链交易状态，确认是否成功打包；
        
    2.  若源链成功，监控目标链回调是否触发，核对 gas 配置与合约逻辑；
        
    3.  若目标链执行失败，自动发起**回滚流程**，将资金退回源链用户地址；
        
    4.  记录失败日志，支持开发者通过 Reactscan（RSC 区块浏览器）排查问题。
        
3.  **开发者建议**：在合约中预设**失败回调逻辑**，并预留足够的跨链 gas，避免因 gas 不足导致的资金锁定。
    

### **问题7：RSC 与传统自动化方案的成本对比**

-   **成本优势**：通过并行化 EVM 大幅降低复杂计算的 gas 成本，且无需支付中心化机器人的服务器托管、运维费用。
    
-   **成本构成**：主要包括合约部署费、跨链回调费、事件订阅费（按需收取）。
    
-   **适用场景**：高频、跨链、复杂逻辑的自动化场景（如跨链套利、自动清算），成本优势显著；简单单链自动化场景（如单池流动性挖矿），成本与传统方案持平。
    

## **三、开发入门核心要点**

### **1\. 开发前置准备**

-   **环境要求**：熟悉 Solidity 语法、EVM 基础，掌握 Git、Node.js 工具链。
    
-   **核心工具**：
    
    -   测试网：Lasna Testnet（Reactive 官方测试网）；
        
    -   开发库：Reactive Library（提供抽象合约与接口）；
        
    -   浏览器：Reactscan（查询 RSC 事件与执行记录）。
        

### **2\. 核心开发步骤（简化版）**

1.  **定义事件订阅**：指定需要监控的链、合约地址、事件签名（如 ERC20 转账事件 `Transfer(address,address,uint256)`）；
    
2.  **编写响应逻辑**：在 Solidity 合约中实现事件触发后的执行逻辑（如跨链转账、套利交易）；
    
3.  **部署合约**：将合约部署至 Reactive Network 主网/测试网；
    
4.  **配置回调**：设置目标链的回调代理地址，确保跨链操作可执行；
    
5.  **测试与上线**：在 Lasna Testnet 完成测试，通过后上线主网。
    

### **3\. 开发者资源**

-   官方文档：[dev.reactive.network](http://dev.reactive.network)（含入门教程、API 参考）；
    
-   示例代码：GitHub Reactive-Smart-Contract-Demos（含 DeFi、NFT 等多场景示例）；
    
-   技术支持：通过 Reactive Network 官方 Discord 或邮件（[support@reactive.network](mailto:support@reactive.network)）获取帮助。
